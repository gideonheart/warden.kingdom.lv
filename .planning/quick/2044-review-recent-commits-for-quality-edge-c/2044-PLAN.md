  <name>Task 1: Code quality review of recent commits</name>
  <files>.planning/quick/2044-review-recent-commits-for-quality-edge-c/CODE-REVIEW.md</files>
  <action>
Review all source code changes from the v3.3 milestone (phases 32-35) and the post-milestone JSON5 fix. Focus on:

1. **Edge cases in the Telegram notification pipeline:**
   - NotificationPoller: What happens if `tmux capture-pane` returns empty output? Does `detectAgentState("")` produce a sane result? Check whether the `working` default causes false-negative suppression.
   - ApprovalCallbackHandler: The `editMessageText` call uses `approval.originalText` which may contain Markdown special characters from the excerpt. If the excerpt contains unmatched backticks or asterisks, the Markdown parse may fail. Assess whether this is a real risk.
   - TelegramBotService: `parseInt(topicId, 10)` will produce `NaN` if topicId is not a numeric string. Check what grammy does with `NaN` as `message_thread_id`.
   - BudgetAlertPoller: `records` Map is in-memory only. On server restart, all budget alert state is lost. First poll after restart may re-fire alerts that were already sent. Assess severity.

2. **Database concerns:**
   - `getBudgetAlertStatus()` does a division `cost_usd / daily_budget_usd * 100`. If `daily_budget_usd` is 0 the WHERE clause filters it out (`WHERE bc.daily_budget_usd > 0`). Confirm this is safe.
   - `notification_config` table uses a single row (id=1). Check whether the migration creates this row or relies on the `INSERT ... ON CONFLICT` in `setNotificationConfig`. If `getNotificationConfig` is called before any `set`, does the default fallback work?

3. **Client-side concerns:**
   - NotificationSettingsPanel: The `defaultValue` + `key` pattern on number inputs is used to force React to re-render with server state on revert. Confirm this is correct React behavior (it is -- key change forces remount).
   - The `onBlur` handler uses `parseInt` which truncates decimals. If user types "2.5" they get 2. Acceptable but document.
   - No error feedback to the user on save failure (silent revert). Assess whether this is adequate UX.

4. **The JSON5 comment stripping fix (2fd1d6e):**
   - The new regex `/"(?:[^"\\]|\\.)*"|\/\/.*$|\/\*[\s\S]*?\*\//gm` correctly handles escaped quotes inside strings. Verify it handles edge cases: empty strings, strings with `\/\/` (escaped slashes), nested quotes.

5. **Tech debt signals:**
   - `detectAgentState()` is 11 lines of regex heuristics shared by NotificationPoller (for Telegram alerts) and the UI (for state chips). No tests exist for this function. It is a single point of fragility.
   - `DatabaseConnection.ts` is 736 lines and growing. It handles instances, recordings, token usage, budget configs, notification configs, rotation configs, and storage stats. It has no decomposition.
   - The Vite build warns about a 674KB chunk. No code splitting is in place.
   - `InstanceTracker.syncWithTmux()` is declared `async` but the `startPeriodicSync` call in index.ts treats it synchronously (no await, no error handling on the promise).

Write findings to CODE-REVIEW.md with sections: Summary, Edge Cases, Tech Debt, Recommendation (severity: info/low/medium/high for each finding).

**Direction constraints:**
- Must address a real operational gap visible in the current codebase or operator workflow
- Must build on existing infrastructure (no greenfield rewrites)
- Must be completable in 3-5 phases (1-3 day effort for Claude)

**Suggested directions to explore (pick 3 or substitute better ones):**

A. **v4.1: Agent Session Analytics Dashboard** — The token_usage table has rich per-agent per-day per-model cost data but the UI only shows a flat table and burn rates. Build a proper analytics view: time-series cost charts, agent efficiency metrics (cost per session, cost per hour of active time), trend lines, anomaly highlighting (agent X cost 3x its normal rate today). This turns raw cost data into operational intelligence.

B. **v4.1: Smart Session Lifecycle** — Sessions are created manually or via auto-record triggers, but there is no intelligence around session lifecycle. Build: auto-restart on crash detection (tmux session disappeared while instance status was 'active'), session timeout policies (auto-stop sessions idle for N hours), session templates (pre-configured agent + project + flags combinations for one-click launch), and a session scheduler (start agent X at 09:00 daily). This moves Warden from passive monitor to active session manager.

C. **v4.1: Warden Plugin/Extension System** — The codebase has grown from 3 to 6 route modules, 3 to 8+ services, and the server index.ts manually wires everything. Build a lightweight plugin architecture: service registry pattern, event bus for cross-service communication (replacing direct imports), plugin manifest for enabling/disabling features (e.g., disable Telegram entirely if bot token not set), and a /api/system/plugins endpoint. This is an infrastructure investment that makes future milestones cheaper.

D. **v4.1: Operator Command Palette & Keyboard Shortcuts** — The dashboard has no keyboard navigation. Build a command palette (Cmd+K) that can: switch between sessions, start/stop agents, toggle views, search session history, and execute saved prompts. Add vim-style keyboard shortcuts for power users. This is a high-frequency UX improvement for the operator who lives in the dashboard.

E. **v4.1: Codebase Quality Foundation** — Address the tech debt findings from Task 1: extract DatabaseConnection into domain-specific repositories, add unit tests for detectAgentState, add code splitting to reduce bundle size, add structured logging (replace console.log/warn/error), add a proper error boundary to the React app. This is a stability investment.

For each, use the format from quick-2043. End with a Recommendation section explaining which to pick and why, considering the findings from CODE-REVIEW.md.
