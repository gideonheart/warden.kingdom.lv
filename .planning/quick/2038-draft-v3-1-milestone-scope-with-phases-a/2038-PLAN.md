**Motivation section:**
Explain why these three features combine naturally: they advance Warden from a monitoring/observation tool to an active operations platform. Agent orchestration lets the operator control agent lifecycle. Token insights help budget and optimize costs. Session recording captures institutional knowledge and enables debugging.

**Feature Area 1: Agent Orchestration Controls (start/stop/restart from dashboard)**
Use requirement ID prefix ORCH-

- ORCH-01: Start a new agent session from the dashboard (select agent from config, optionally specify project/working dir, spawns tmux session with Claude Code)
- ORCH-02: Stop a running agent session gracefully (send SIGTERM to Claude process inside tmux, wait, then kill tmux session)
- ORCH-03: Restart a stopped/errored agent session (stop then start with same config)
- ORCH-04: Agent session status lifecycle (running -> stopping -> stopped, with UI state transitions)
- ORCH-05: Safety guards (confirmation dialog before stop/restart, prevent double-start)

Note existing infrastructure: GSD ControlsTab already has spawn dispatch capability via POST /api/gsd/spawn. TmuxSessionManager already wraps tmux CLI calls. InstanceTracker already tracks session lifecycle. Build ON these, do not rebuild.

Priority assignments: ORCH-01 through ORCH-04 are P1, ORCH-05 is P1 (safety-critical).

**Feature Area 2: Enhanced Token Insights (burn rate, budget alerts, model comparison)**
Use requirement ID prefix TOKN-

- TOKN-10: Burn rate calculation (tokens/hour, cost/hour over sliding window — last 1h, 4h, 24h)
- TOKN-11: Budget alerts — set daily/weekly cost threshold per agent, visual warning when approaching (80%) and exceeded (100%)
- TOKN-12: Model comparison view — side-by-side cost-per-task breakdown by model (sonnet vs opus vs haiku) across agents
- TOKN-13: Cost projection — at current burn rate, estimated daily/weekly spend
- TOKN-14: Export token usage data as CSV for external analysis

Note existing infrastructure: SessionUsageReader already scans JSONL files with per-model pricing. TokenUsageView exists with cache columns and Scan Now. token_usage SQLite table stores daily aggregates. Build ON these.

Priority: TOKN-10 and TOKN-11 are P1. TOKN-12 and TOKN-13 are P2. TOKN-14 is P2.

**Feature Area 3: Session Recording & Replay**
Use requirement ID prefix REC-

- REC-01: Capture terminal output stream to a recording file (lightweight format — timestamped chunks, not full video)
- REC-02: Recording toggle per session (start/stop recording from UI)
- REC-03: Replay a recorded session in a read-only xterm.js terminal at original speed or 2x/4x/8x
- REC-04: Recording library/browser — list past recordings with session metadata (agent, project, date, duration)
- REC-05: Auto-record option (configurable: always record, never record, record on permission prompt detection)

Note: Standard approach is asciicast/asciinema format (JSON with timestamped events). The `xterm-addon-serialize` addon can capture terminal state. Consider recording at the PTY tap level (TerminalStreamService already has setImmediate tap from Phase 11).

Priority: REC-01 and REC-02 are P1. REC-03 and REC-04 are P2. REC-05 is P3.

**Phase Breakdown (starting at Phase 21):**

Phase 21 — Agent Lifecycle Controls
- Goal: Operator can start, stop, and restart agent sessions directly from the Warden dashboard with safety confirmations.
- Requirements: ORCH-01, ORCH-02, ORCH-03, ORCH-04, ORCH-05
- Depends on: Phase 20 (existing infrastructure only)
- Acceptance criteria (5):
  1. Operator can start a new agent session by selecting an agent and clicking Start — tmux session appears in tab bar within 15 seconds
  2. Operator can stop a running session — confirmation dialog appears, session transitions to "stopping" then "stopped" state
  3. Operator can restart a stopped session — equivalent to stop + start with same config
  4. Session status badges in InstanceTabBar reflect lifecycle state (running/stopping/stopped) in real time
  5. Double-start prevention: Start button disabled/hidden for agents with active sessions

Phase 22 — Token Burn Rate & Budget Alerts
- Goal: Operator sees real-time cost velocity and gets warned before agents blow through budget thresholds.
- Requirements: TOKN-10, TOKN-11, TOKN-13
- Depends on: Phase 18 (SessionUsageReader, token_usage table)
- Acceptance criteria (4):
  1. TokenUsageView shows burn rate (cost/hour) for each agent with sliding window selector (1h/4h/24h)
  2. Budget threshold configuration stored per-agent (daily limit in USD) with visual warning at 80% and alert at 100%
  3. Cost projection line shows estimated daily spend at current burn rate
  4. Budget alerts visible as colored badge on the History nav tab when any agent exceeds threshold

Phase 23 — Token Analytics & Export
- Goal: Operator can compare model costs across agents and export data for external analysis.
- Requirements: TOKN-12, TOKN-14
- Depends on: Phase 22 (burn rate data structures)
- Acceptance criteria (3):
  1. Model comparison view shows cost breakdown by model variant (sonnet/opus/haiku) per agent as bar chart or table
  2. Export button downloads CSV with columns: date, agent, model, input_tokens, output_tokens, cache_read, cache_write, cost_usd
  3. Export covers full date range in token_usage table (not just visible page)

Phase 24 — Session Recording & Replay
- Goal: Operator can record terminal sessions and replay them later at variable speed in a read-only terminal.
- Requirements: REC-01, REC-02, REC-03, REC-04
- Depends on: Phase 20 (TerminalStreamService PTY tap)
- Acceptance criteria (5):
  1. Record button in terminal view header starts capturing PTY output with timestamps to a recording file
  2. Stop recording button finalizes the file — recording appears in the recording library
  3. Recording library shows list of past recordings with agent name, project, date, duration
  4. Clicking a recording opens a read-only xterm.js terminal that replays the session
  5. Playback speed controls (1x, 2x, 4x, 8x) work during replay

Phase 25 — Recording Automation (Stretch)
- Goal: Recordings can be triggered automatically based on configurable rules.
- Requirements: REC-05
- Depends on: Phase 24
- Acceptance criteria (2):
  1. Settings panel offers auto-record options: always, never, on permission prompt
  2. Auto-record triggers immediately when configured condition is met

**Out of Scope for v3.1:**
- Multi-user access control or role-based permissions
- Agent creation/deletion from dashboard (agents are defined in openclaw.json)
- Remote server agent management (single-server only)
- Video recording of terminal (use lightweight asciicast format)
- Real-time streaming replay (only completed recordings)
- Mobile UI improvements (deferred from v2.0)
- Telegram alert forwarding (was P3 in v3.0, still deferred)
- detectAgentState() rewrite (deferred, regex heuristics remain)

**Dependencies and Risks:**
- Phase 21 (Agent Orchestration) depends on tmux CLI reliability and spawn.sh patterns from v2.1
- Phase 22-23 (Token Insights) depends on SessionUsageReader JSONL format stability
- Phase 24 (Recording) needs storage management strategy — recordings could grow large
- Risk: Agent start/stop involves process lifecycle management — error handling must be robust
- Risk: Claude Code session JSONL format is not a documented API — could change between versions
- Risk: Recording storage on single server could exhaust disk — need size limits or rotation

**Technical Notes:**
Include brief notes on:
- Agent start: Reuse spawn.sh pattern from v2.1 (POST /api/gsd/spawn returns 202)
- Agent stop: `tmux send-keys C-c` followed by `tmux kill-session` after grace period
- Burn rate: Compute from token_usage table with time-windowed SQL queries
- Recording format: asciicast v2 (JSON lines with [time, type, data] tuples) — industry standard, compatible with asciinema player
- Storage: data/recordings/ directory, one file per recording, SQLite metadata table

**Milestone Acceptance Criteria (7 items):**
1. Operator can start a new agent session from the dashboard without touching the CLI
2. Operator can stop/restart sessions with safety confirmation dialogs
3. Token burn rate is visible per agent with configurable budget alerts
4. Model cost comparison available for spend optimization decisions
5. Terminal sessions can be recorded and replayed at variable speed
6. All features work in production build (npm run build && npm start)
7. No regressions in existing E2E tests

The document should be approximately 300-400 lines, well-structured with markdown headers, horizontal rules between sections, and the same professional tone as v3.0-SCOPE.md.
