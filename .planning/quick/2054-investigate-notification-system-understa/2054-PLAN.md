**1. Current Notification Architecture Map**

Document all three existing notification channels with their complete data flow:

A. **Telegram Permission Prompt Notifications** (server-side):
   - Flow: NotificationPoller (10s poll) -> tmux capture-pane -> strip ANSI -> detectAgentState() -> NotificationDeduplicator (state transition + cooldown check) -> TelegramBotService.sendToTopic() -> Telegram Bot API -> agent's Telegram group/topic
   - Config: `notification_config` SQLite table (permissionAlertsEnabled, permissionCooldownMs)
   - Dependencies: openclaw.json (bot token, topic mappings per agent), tmux sessions
   - Known issue from STATE.md: "NotificationPoller polls stopped sessions (dead capture-pane calls)"

B. **Telegram Budget Alert Notifications** (server-side):
   - Flow: BudgetAlertPoller (10s poll) -> database.getBudgetAlertStatus() -> level escalation/cooldown check -> TelegramBotService.sendToTopic() -> Telegram Bot API
   - Config: `notification_config` table (budgetAlertsEnabled, budgetCooldownMs), `budget_alert_states` table (persistent dedup across restarts)
   - Dependencies: openclaw.json (bot token, topic mappings), token_usage database

C. **Telegram Crash Notifications** (server-side, event-driven):
   - Flow: InstanceTracker.onCrashDetected -> inline handler in index.ts -> TelegramBotService.sendToTopic()
   - Not configurable (always fires if bot token + topic mapping exist)

D. **Browser Budget Notifications** (client-side):
   - Flow: useBudgetAlerts hook -> budgetAlertLevel state -> useBrowserNotifications hook -> Notification API (ref-based transition detection + tag dedup)
   - Config: localStorage opt-in ('warden:notifications-enabled'), browser permission API
   - UI: Bell icon toggle in TerminalView header, NotificationSettingsPanel in History view
   - Limitation: Only fires for budget alerts, NOT for permission prompts (those go via Telegram only)

**2. What Does NOT Exist Today**

Identify the gaps explicitly:
- No PWA / web app manifest (no "install to home screen" capability)
- No Service Worker (required for background push notifications)
- No Web Push API integration (server-side VAPID key generation, subscription management)
- No push subscription storage (no database table for push endpoints)
- Browser notifications only work while the tab is open and in focus
- No native mobile app (no React Native, Capacitor, etc.)
- Permission prompt notifications are Telegram-only (no browser notification for them)

**3. Options for Native OS/Phone Notifications**

Present 3-4 concrete options with tradeoffs:

**Option A: Web Push Notifications (PWA)**
- What: Add a service worker + Web Push API + VAPID keys to make Warden a Progressive Web App
- How it works: Server generates VAPID key pair, client subscribes via PushManager API, server stores subscription endpoint, server sends push events via web-push library
- Pros: Works on desktop (Chrome, Firefox, Edge) and Android Chrome, no app store needed, "install to home screen" support
- Cons: Does NOT work on iOS Safari (no Web Push support without being installed as PWA first, and even then limited), requires HTTPS in production (localhost works for dev), adds web-push npm dependency
- Effort: Medium (2-3 phases: service worker + push subscription + server-side push sender + notification settings UI update)
- iOS note: As of iOS 16.4+, Web Push works ONLY if the site is added to home screen as a PWA first

**Option B: Telegram as the Primary Mobile Channel (already exists)**
- What: Telegram is already sending notifications to the user's phone via the Telegram app
- How it works: Already implemented -- NotificationPoller and BudgetAlertPoller send to Telegram topics
- Pros: Already built and working, works on all platforms (iOS, Android, desktop), reliable push via Telegram infrastructure
- Cons: Requires Telegram app, notifications go to a group topic (not a direct message), user must have Telegram open/configured
- Effort: Zero (already done)
- Enhancement opportunity: Add direct message support (not just topics), or add a summary/digest notification

**Option C: ntfy.sh (Self-hosted Push Relay)**
- What: Use ntfy.sh (open source, self-hostable) as a push notification relay
- How it works: Server POSTs to ntfy.sh topic URL, user subscribes to topic via ntfy mobile app or browser
- Pros: Free, open source, self-hostable (no paid dependency), works on iOS and Android via ntfy app, simple HTTP POST API
- Cons: Requires user to install ntfy app, one more external service to maintain if self-hosted
- Effort: Low (1 phase: add ntfy sender service alongside TelegramBotService, config UI)

**Option D: Unified Push + Gotify (Self-hosted Alternative)**
- What: Use Gotify (self-hosted push server) for notifications
- How it works: Similar to ntfy but with its own client apps
- Pros: Fully self-hosted, no external dependencies
- Cons: Less widely known, requires running another server process, smaller ecosystem
- Effort: Low-Medium

**4. Recommendation**

Based on the project's constraint of "no paid-subscription dependencies" and the existing Telegram integration:
- Telegram already covers the phone notification use case for the current operator
- Web Push (Option A) would be the most standard way to add native browser notifications that work when the tab is closed
- ntfy.sh (Option C) is the lowest-effort path for phone push if Telegram is not sufficient
- The most impactful quick win would be extending browser notifications to also fire for permission prompts (currently Telegram-only), which requires no new infrastructure

**5. Quick Win: Browser Notifications for Permission Prompts**

Document what it would take to add browser notifications for permission prompts alongside the existing Telegram flow:
- Server already emits Socket.IO events -- could add a `notification:permission_prompt` event
- Client already has useBrowserNotifications infrastructure -- extend it to listen for permission prompt events
- No new dependencies needed, ~1 quick task of effort
