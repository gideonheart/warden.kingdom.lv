Review the 4 fixes in commit 6dd185d and produce a structured review document at the files path above. The review MUST cover the following specific findings (discovered during planning analysis) plus any additional findings from deeper inspection:

**Fix 1 — Callback refs in useTerminalSocket (lines 50-57):**
- RISK (medium): Refs are assigned during render body (`onTerminalOutputRef.current = onTerminalOutput` on lines 55-57). This is technically a side-effect during render. React concurrent mode may discard a render and replay it, causing the ref to be updated for a render that was never committed. In practice this is widely used and benign (the ref just gets the same value again on replay), but it violates React's "render must be pure" rule. The strictly safe alternative is `useLayoutEffect` for ref sync. Assess whether this matters for this codebase (React 19, no StrictMode double-render observed).
- EDGE CASE: The `[isConnected]` effect (lines 60-81) has an empty cleanup return with comment "Nothing to clean up here". The timer cleanup relies on a separate `[]` unmount effect (lines 84-91). If React batches state updates such that isConnected transitions false->true->false rapidly, the timer logic could have an edge where the timer starts, is cleared, then a new timer starts, all within one batch — verify this is safe.
- OBSERVATION: `showConnectingOverlay` initial state is `false` while `isConnected` initial state is also `false`. This means on very first mount there is a 500ms window with NO visual feedback before the overlay appears. On slow connections (mobile 3G), the user may see a blank terminal area for 500ms before any spinner.

**Fix 2 — Overlay delay (OVERLAY_DELAY_MS):**
- RISK (low): The 500ms delay value is hardcoded. PTY exit triggers a 2000ms setTimeout before reconnectGeneration++. So the reconnect cycle is: socket disconnect -> 2s wait -> new socket -> connect event. Total time > 2s, which means the overlay WILL show for ~1.5s during PTY exit auto-reconnect. The overlay delay only helps for sub-500ms reconnects (network hiccups). Document this — the PTY-exit reconnect still shows the overlay, which may be acceptable or may need the timer increased.

**Fix 3 — Ref mutation outside setState in useSessionSelection:**
- RISK (medium): `selectedSessionName` is read from closure on line 95 (`const currentSession = selectedSessionName`) but is intentionally excluded from the effect's dependency array (line 121). This creates a stale closure: if the user calls `selectSession('X')` and a poll arrives in the same React batch, the effect reads the pre-selectSession value. The `if (selectedSession !== currentSession)` guard on line 117 could then set state to an incorrect value. Walk through this scenario step by step:
  1. User clicks session-B (selectSession sets state to session-B)
  2. Poll arrives in same batch (effect runs with currentSession = session-A, the stale value)
  3. resolveSessionFallback('session-A', ['session-A','session-B'], 0, false) returns {selectedSession: 'session-A', resetMissCount: true}
  4. 'session-A' !== 'session-A' is false, so setState is NOT called — user's manual selection to session-B survives
  Actually, React processes effects after commits, so the stale value would be from the render where the poll triggered, not the render where selectSession triggered. This needs careful analysis of whether React batches these correctly or if there is a real race.
- OBSERVATION: The `consecutiveMissCountRef` update happens BEFORE setState. If the ref increment to 1 is followed by a re-render from another source (not this effect), and then another poll arrives, the miss count could be read as 1 when it should still be 0. Trace this path.

**Fix 4 — Dimension guard in TerminalView:**
- RISK (low): The 20px threshold is arbitrary. A very small terminal (e.g., narrow sidebar embed if ever added) could be permanently skipped. This is acceptable for now since TerminalView always lives in a full-width container.
- EDGE CASE: After keyboard dismissal on iOS, the container grows back to full size. The next resize event triggers refitTerminal. But since the guard skips degenerate sizes, what happens if the LAST resize event fires when the container is mid-transition and below 20px? The terminal stays at stale dimensions until the next resize event. Verify that iOS Safari fires at least one resize event after layout is final. (iOS Safari does fire visualViewport resize on keyboard dismiss — this should work.)

**Test quality assessment:**
- useTerminalSocket.test.ts: All 5 tests are structural/documentation — they assert `expect(true).toBe(true)`, `expect(500).toBeGreaterThan(0)`, and `expect(invariants).toHaveLength(7)`. NONE of these would catch a regression. They are valuable as documentation but provide ZERO behavioral protection. A follow-up task should add real behavioral tests using @testing-library/react to verify: (a) socket effect only runs when sessionName changes, (b) overlay does not appear within 500ms of disconnect, (c) overlay appears after 500ms.
- useSessionSelection.test.ts: The `resolveSessionFallback` tests (8 tests) are genuinely behavioral and excellent. The `idempotency` tests (2 tests) are behavioral. The `ref mutation invariant` test (1 test) is behavioral but only tests the pure function, not the hook's ref mutation behavior.
- useActiveInstances.test.ts: The `computeInstanceSignature` tests (6 original + 6 new) are all genuinely behavioral and provide strong regression protection for the signature dedup layer.

**Overall assessment:** Fixes are architecturally sound and address the correct root causes. The callback-ref pattern is the industry standard for this problem. The overlay delay is a pragmatic UX improvement. The ref-mutation fix correctly addresses a React anti-pattern. The dimension guard is a sensible defensive measure.
