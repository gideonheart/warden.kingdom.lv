<objective>
Stabilize terminal rendering and session selection lifecycle so that polling does not disrupt active terminal streams.

Purpose: The 5-second instance poll creates new array references on every cycle, causing App.tsx to re-render. Render-body setState calls for session selection/fallback fire on each re-render, which can flicker the selected session name and cause TerminalView (keyed by session name) to remount, triggering Socket.IO reconnection and terminal corruption. Fix the root causes: (1) make poll skip setState when data is logically unchanged, (2) move all selection logic into a dedicated hook with useEffect guards, (3) add hysteresis for session disappearance.

Output: Stabilized polling + selection + terminal lifecycle, unit tests, passing build.
</objective>

**Step 1c: Create `src/client/hooks/useSessionSelection.ts` — the session selection policy hook.**

This hook encapsulates ALL session selection logic currently scattered in App.tsx render body.

Interface:
```ts
interface UseSessionSelectionParams {
  activeInstances: AgentInstance[];
  isLoading: boolean;
  initialSessionName: string | null;
}

interface UseSessionSelectionResult {
  selectedSessionName: string | null;
  selectSession: (sessionName: string) => void;
  clearSelection: () => void;
}
```

Internal behavior:
1. **State:** `selectedSessionName` managed via `useState`, initialized from `initialSessionName` param.
2. **Auto-select first session:** `useEffect` — when `isLoading` is false and `selectedSessionName` is null and `activeInstances.length > 0`, set to `activeInstances[0].tmuxSessionName`. Guard: only fire when these conditions transition (use a ref to track if auto-select has fired for this loading cycle).
3. **Fallback on session disappearance with hysteresis:** `useEffect` — when `selectedSessionName` is set but not found in `activeInstances` and `isLoading` is false:
   - Increment a `consecutiveMissCountRef` (useRef, init 0).
   - If miss count is less than 2, do NOT change selection (transient miss tolerance).
   - If miss count is 2 or more, fall back to `activeInstances[0]?.tmuxSessionName ?? null`.
   - Reset `consecutiveMissCountRef` to 0 whenever the selected session IS found in `activeInstances`.
4. **User explicit selection:** `selectSession(name)` sets state directly and resets miss counter to 0.
5. **clearSelection:** sets `selectedSessionName` to null.

Key: This hook must NOT call setState during render. All mutations happen inside useEffect or event handlers.

Export the pure function `resolveSessionFallback(params)` separately so it can be unit-tested without React:
```ts
export function resolveSessionFallback(
  currentSession: string | null,
  activeSessionNames: string[],
  consecutiveMisses: number,
  isLoading: boolean,
): { selectedSession: string | null; resetMissCount: boolean } {
  // If loading, keep current
  if (isLoading) return { selectedSession: currentSession, resetMissCount: false };
  // If no current, auto-select first
  if (!currentSession && activeSessionNames.length > 0) {
    return { selectedSession: activeSessionNames[0], resetMissCount: true };
  }
  // If current found, keep it
  if (currentSession && activeSessionNames.includes(currentSession)) {
    return { selectedSession: currentSession, resetMissCount: true };
  }
  // Current missing — hysteresis
  if (currentSession && !activeSessionNames.includes(currentSession)) {
    if (consecutiveMisses < 1) {
      // First miss — keep current, do not reset
      return { selectedSession: currentSession, resetMissCount: false };
    }
    // Second consecutive miss — fallback
    return {
      selectedSession: activeSessionNames[0] ?? null,
      resetMissCount: true,
    };
  }
  return { selectedSession: currentSession, resetMissCount: false };
}
```

The hook calls `resolveSessionFallback` inside a `useEffect` that depends on a stable stringified key for activeInstances (e.g., `activeInstances.map(i => i.tmuxSessionName).join(',')`) and `isLoading`, so it only re-runs when the session list actually changes.
