Review the three phase-37 implementation commits (6e1d4bb, a58f3ec, 54e1d7c) by reading the current state of all modified files. Evaluate for the following categories:

**Edge Cases to Check:**
1. `preExistingSessionNames` is computed on EVERY sync call (not just initial) -- this Set is populated before upserts, used to guard `started` event logging. Evaluate: is it wasteful on subsequent syncs since `initialSyncComplete` guards the insert anyway? (Answer: minor -- the Set construction is harmless since `initialSyncComplete` gates usage; the real guard works. No bug, just unnecessary work on every poll.)

2. `detectCrashesAndMarkStopped` calls `database.listAllInstances()` which returns ALL instances (including stopped/error). The status filter at lines 90-96 handles this, but the query fetches potentially hundreds of old stopped instances on every 10s poll. Evaluate performance at scale.

3. `GET /api/lifecycle-events` endpoint: `parseInt(request.query.limit, 10)` will produce `NaN` if a non-numeric string is passed (e.g., `?limit=abc`). `NaN` is then passed to SQLite `LIMIT NaN` -- evaluate what happens (SQLite treats NaN as 0, so it returns 0 rows instead of the default 50). Same issue for `offset`.

4. Force-kill endpoint (`POST /api/instances/:id/force-kill`) does NOT log a lifecycle event -- already flagged in VERIFICATION.md as a warning. Confirm this is the only gap.

5. `markMissingSessionsStopped()` is still in DatabaseConnection but no longer called from syncWithTmux -- it's dead code from the crash detection refactor. The first sync now skips crash detection entirely (`!initialSyncComplete`), but also skips `markMissingSessionsStopped`. This means sessions that were 'active' in the DB but whose tmux sessions died while the server was down will NOT be marked stopped until the SECOND poll cycle. Evaluate whether this 10s gap matters.

6. `onCrashDetected` callback is set AFTER `startPeriodicSync()` in index.ts. Since `startPeriodicSync()` calls `syncWithTmux()` immediately (line 20-21), there's a brief moment during the first sync where `onCrashDetected` is null. But since `initialSyncComplete` is false on first sync, crash detection is skipped anyway -- so this is safe. Confirm.

7. The Telegram crash message includes `*${instance.agentId}*` with Markdown bold. If agentId contains Markdown special characters (unlikely but possible), the message could fail. Phase 36 fixed Markdown escaping for pane excerpts -- verify whether agentId needs the same treatment.

8. `missedPollCounts` Map is never persisted -- it resets on server restart. Combined with `initialSyncComplete=false`, this means a crashed session that was mid-grace-period before restart will get a fresh 2-poll grace period after restart. This is actually desirable behavior (avoids false alerts on restart).

**Tech Debt to Catalogue:**
- Dead code: `markMissingSessionsStopped()` no longer called from InstanceTracker
- Performance: `listAllInstances()` fetches all rows every 10s; consider adding a `listNonStoppedInstances()` query
- Missing lifecycle event: force-kill endpoint
- NaN handling: lifecycle-events API parseInt without validation
- Redundant computation: preExistingSessionNames Set built on every sync
