The new session runs `claude --dangerously-skip-permissions` in the agent's working_directory from agent-registry. Unlike the existing start endpoint, this does NOT block on duplicate active sessions — that is the entire point: running parallel sessions for the same agent.

Purpose: Allows operators to run quick tasks simultaneously without stopping or overriding the current agent session.
Output: Working "New Session" button that creates discoverable tmux sessions.

  <files>src/server/routes/instanceRoutes.ts</files>
  <action>
Add a new `POST /api/instances/spawn` endpoint in instanceRoutes.ts, modeled after the existing `POST /api/instances/start` but WITHOUT the duplicate-session guard (`findActiveByAgentId` check that returns 409).

The endpoint accepts `{ agentId: string, projectPath?: string }` in the request body. Flow:

1. Validate `agentId` is a non-empty string (same as start endpoint).
2. Look up agent config from `openClawConfigReader.getAgents()` to get `agentName`.
3. Resolve `projectPath`: if provided in body, use it. Otherwise, check `gsdRegistryService.getAgent(agentId)` for `working_directory`. Fall back to openclaw.json `workspace` field (same resolution logic as start endpoint).
4. Derive `projectSlug` from `path.basename(projectPath)`.
5. Build session name via `tmuxSessionManager.buildSessionName(agentId, projectSlug)` — this generates `{agentId}-{projectSlug}-{shortUuid}` with a random 4-char UUID, so it will NOT collide with existing sessions.
6. Pre-register instance via `database.upsertInstance(...)` with status 'starting'.
7. Fire-and-forget `tmuxSessionManager.createSessionWithClaude(sessionName, projectPath)` — on success set status to 'active', on failure set to 'error'.
8. Return 202 with `{ message: 'Spawning additional session', instance }`.

DO NOT include the `instanceTracker.findActiveByAgentId()` duplicate guard that the start endpoint has. That guard is intentionally omitted because spawning parallel sessions is the whole point.

Place the endpoint right after the existing `POST /api/instances/start` route for logical grouping.
  </action>

  </verify>
  <done>POST /api/instances/spawn endpoint exists, accepts agentId, creates a new tmux session with claude running, and returns 202 even when the agent already has an active session.</done>

   ```typescript
   const handleSpawnSession = useCallback(async () => {
     const instance = selectedInstanceRef.current;
     if (!instance) return;
     try {
       const response = await fetch('/api/instances/spawn', {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify({ agentId: instance.agentId, projectPath: instance.projectPath }),
       });
       if (!response.ok) {
         console.error(`Spawn failed: ${response.statusText}`);
       }
       refetch();
     } catch (error) {
       console.error('Error spawning session:', error);
     }
   }, [refetch]);
   ```
