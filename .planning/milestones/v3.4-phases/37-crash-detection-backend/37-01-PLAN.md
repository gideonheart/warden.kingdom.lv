**1. Rewrite InstanceTracker crash detection logic:**

Add a private `missedPollCounts` Map to track consecutive missed polls:
```typescript
private missedPollCounts: Map<string, number> = new Map(); // key: tmuxSessionName
```

Add a constant for the grace period:
```typescript
const CRASH_GRACE_POLLS = 2; // Require 2 consecutive missed polls (~20s) before declaring crash
```

**2. Modify `syncWithTmux()` to detect crashes before marking sessions stopped:**

Replace the call to `database.markMissingSessionsStopped(activeSessionNames)` with a new method `detectCrashesAndMarkStopped(activeSessionNames)` that does the following:

a) Get all instances with status IN ('active', 'idle') whose tmux_session_name is NOT in `activeSessionNames`.

b) For each missing session:
   - If `instance.status === 'stopping'`: this is an operator-initiated stop. Mark as 'stopped', log lifecycle event with `eventType: 'stopped'`, `stopReason: 'operator-stop'`. Remove from missedPollCounts.
   - If `instance.status === 'active'` or `instance.status === 'idle'`:
     - Increment missedPollCounts for this session name
     - If missedPollCounts < CRASH_GRACE_POLLS: skip (grace period, do NOT mark stopped yet)
     - If missedPollCounts >= CRASH_GRACE_POLLS: this is a CRASH.
       - Update instance status to 'stopped' (there is no 'crashed' status — crash is recorded as a lifecycle event)
       - Log lifecycle event with `eventType: 'crashed'`, `stopReason: 'crash'`, `outcome: 'detected'`
       - Calculate `uptimeSecs` from `instance.createdAt` to now
       - Extract `projectSlug` from `instance.projectPath` (last path segment via `path.basename()`)
       - Store `lastKnownState: instance.status` (the status before crash)
       - Remove from missedPollCounts
   - Sessions in 'starting' or 'error' or 'stopped' status: skip (handled elsewhere or already terminal)

c) For sessions that ARE in activeSessionNames: clear their missedPollCounts entry (they're alive).

**3. Log lifecycle events for session starts (in upsert flow):**

In the `syncWithTmux()` upsert loop, after `database.upsertInstance(...)`, check if the instance was just created (new session appeared for the first time) by comparing before/after. If newly created, insert a lifecycle event with `eventType: 'started'`.

Implementation approach: Before the upsert loop, collect existing session names from DB. After upsert, if session name was NOT in the pre-existing set, log a 'started' event. But be careful on server restart — if the server restarts and re-discovers running sessions, these should NOT generate 'started' events. To handle this:
- Add a private `initialSyncComplete: boolean = false` flag.
- On the FIRST sync call, just populate missedPollCounts and mark `initialSyncComplete = true` without logging 'started' events (these are pre-existing sessions discovered on startup).
- On subsequent sync calls, log 'started' events for genuinely new sessions.

**4. Handle the `reconcileTransitionalStates` 'stopping' timeout path:**

When a 'stopping' session exceeds `STOPPING_FORCE_KILL_THRESHOLD_MS` and is force-killed, log a lifecycle event with `eventType: 'stopped'`, `stopReason: 'operator-stop'`, `outcome: 'force-killed'`.

When a 'stopping' session exits cleanly (session disappears within grace period), log a lifecycle event with `eventType: 'stopped'`, `stopReason: 'operator-stop'`, `outcome: 'graceful'`.

When a 'starting' session times out (never appeared), log a lifecycle event with `eventType: 'stopped'`, `stopReason: 'start-failed'`, `outcome: 'timeout'`.

**5. Expose a method for crash notification (used by Plan 02):**

Add an `onCrashDetected` callback property that can be set externally:
```typescript
public onCrashDetected: ((event: { instance: AgentInstance; uptimeSecs: number; projectSlug: string }) => void) | null = null;
```

Call this callback in the crash detection path (step 2b) after logging the lifecycle event. This allows Plan 02 to wire up Telegram notifications without modifying InstanceTracker further.

**6. Update instanceRoutes.ts stop endpoint:**

The existing stop API already sets status to 'stopping' before calling gracefulStopSession(). This is the graceful stop marker that CRSH-01 requires. No changes needed to the stop API itself.

However, after the stop completes (both graceful and forced paths), insert a lifecycle event:
- In the `!sessionExists` early-return path: log `eventType: 'stopped'`, `stopReason: 'operator-stop'`, `outcome: 'already-gone'`
- After `gracefulStopSession()`: log `eventType: 'stopped'`, `stopReason: 'operator-stop'`, `outcome: exitedGracefully ? 'graceful' : 'force-killed'`

Calculate uptimeSecs from `instance.createdAt` to now. Extract projectSlug via `path.basename(instance.projectPath)`.

Import `path` and `database` (database is already imported) in instanceRoutes.ts.

**Important: Do NOT modify `markMissingSessionsStopped()` in DatabaseConnection.** Instead, the new logic in InstanceTracker replaces the call to it with the crash-aware version. Keep markMissingSessionsStopped() available for backward compatibility but it will no longer be called from syncWithTmux() directly — the new crash detection method handles that flow.
