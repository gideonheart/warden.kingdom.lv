**AutoRestartService (src/server/services/AutoRestartService.ts) — NEW FILE:**

Create a new service class with the following:

- Constants:
  - `RESTART_DELAY_MS = 7_000` (7 seconds delay before spawning — within the 5-10s range from CONTEXT decisions)
  - `MAX_RESTARTS_PER_HOUR = 3` (storm threshold)
  - `STORM_WINDOW_MS = 60 * 60 * 1000` (1 hour sliding window)

- State:
  - `restartTimestamps: Map<string, number[]>` — per-agent array of restart timestamps for sliding window tracking (key: agentId)

- Constructor dependencies (injected or imported singletons):
  - `database` from DatabaseConnection
  - `tmuxSessionManager` from TmuxSessionManager
  - `telegramBotService` from TelegramBotService
  - `openClawConfigReader` from OpenClawConfigReader

- `async attemptRestart(crashedInstance: AgentInstance, uptimeSecs: number, projectSlug: string): Promise<void>`:
  1. Read the agent's restart policy via `database.getRestartPolicy(crashedInstance.agentId)`
  2. If `crashRestartMode === 'none'`, log and return early (no restart)
  3. Check rate limit via `isRateLimited(crashedInstance.agentId)`:
     - If rate limited, call `handleStormDetected(crashedInstance.agentId)` and return
  4. Wait `RESTART_DELAY_MS` via `await new Promise(resolve => setTimeout(resolve, RESTART_DELAY_MS))`
  5. Derive `projectPath` from `crashedInstance.projectPath`, `projectSlug` from `path.basename(projectPath) || agentId`
  6. Build new session name via `tmuxSessionManager.buildSessionName(agentId, projectSlug)`
  7. Pre-register instance in DB: `database.upsertInstance(...)` with status 'starting'
  8. Try `await tmuxSessionManager.createSessionWithClaude(agentId, projectSlug, projectPath)`:
     - On success: update status to 'active', log lifecycle event with eventType 'auto-restarted', outcome 'success'
     - On failure: update status to 'error', log lifecycle event with eventType 'auto-restarted', outcome 'failed', include error message in stopReason
  9. Record restart timestamp for rate limiting: `this.recordRestart(agentId)`
  10. If policy is 'once', the policy stays as 'once' — it means "restart once per crash event", NOT "restart once ever". No policy change needed after successful restart.
  11. Log to console: `[AutoRestart] Restarted ${agentId}: ${newSessionName}`

- `private isRateLimited(agentId: string): boolean`:
  1. Get timestamps array for agent, filter to only those within STORM_WINDOW_MS of now
  2. Update the stored array (prune old entries)
  3. Return `filteredTimestamps.length >= MAX_RESTARTS_PER_HOUR`

- `private recordRestart(agentId: string): void`:
  1. Get or create timestamps array for agent
  2. Push `Date.now()`
  3. Prune entries older than STORM_WINDOW_MS

- `private async handleStormDetected(agentId: string): Promise<void>`:
  1. Log warning: `[AutoRestart] Storm detected for ${agentId} — disabling auto-restart`
  2. Call `database.markStormDisabled(agentId)` — sets crash_restart_mode to 'none' and storm_disabled_at to current timestamp
  3. Log lifecycle event: eventType 'auto-restarted', outcome 'storm-disabled', stopReason 'restart-storm'
  4. Send Telegram storm alert:
     - Look up topic mapping via `openClawConfigReader.getTopicMappings()`
     - Format message: "Warning: {agentId} restarted {count} times in 1h — auto-restart disabled. Check dashboard."
     - Send via `telegramBotService.sendToTopic(groupId, topicId, text)`
     - Wrap in try/catch — notification failure must never block storm handling
  5. Need the crashed instance's sessionId for lifecycle event — pass it as parameter or get from last crashed event

- Export singleton: `export const autoRestartService = new AutoRestartService()`

**Toast notification for auto-restarts (src/client/App.tsx):**

The dashboard already polls `/api/instances` via `useActiveInstances` hook every few seconds. When a new session appears with a 'starting' or 'active' status for an agent that also has a recently-crashed (stopped) session visible, the operator should see a brief toast notification.

Implementation approach — lightweight inline toast (no new library dependency):
1. Add a `toastMessages` state: `useState<Array<{ id: string; text: string }>>([])`
2. Add a `previousInstanceIdsRef` to track known instance IDs across polls
3. In a `useEffect` that watches `instances`:
   - Compare current instance IDs with previousInstanceIdsRef
   - For each NEW instance that wasn't in the previous set:
     - Check if there's also a 'stopped' instance for the same agentId in the current list
     - If so, this is likely an auto-restart: add a toast message like `"{agentId} auto-restarted — new session active"`
   - Update previousInstanceIdsRef with current IDs
4. Toast display: render a fixed-position container (bottom-right) showing active toasts
   - Each toast auto-dismisses after 5 seconds via setTimeout
   - Styled with `warden-*` theme tokens: `bg-warden-panel`, `border-warden-accent`, `text-warden-text`
   - Compact, non-blocking, does not cover terminal content
5. Remove toast from state after dismiss timeout

This is a simple, self-contained addition — no new components/files needed for a basic toast.
