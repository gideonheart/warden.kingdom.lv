## Motivation

v3.0 made Warden a complete operator workstation for _monitoring_ — permission prompts surface as badges, context pressure is visible in the terminal header, keyboard shortcuts enable fast navigation, and terminal search finds anything in 50,000 lines of scrollback. The operator can see everything and find everything.

But Warden is still a read-only window into agent activity. The operator sees that an agent is idle or stuck, but cannot restart it from the dashboard. The operator sees that an agent consumed $47 in tokens today, but cannot tell whether that spend is trending up or whether a budget threshold has been crossed. And when something interesting happens in a session — a complex debugging sequence, a novel approach to a task — it scrolls away and is lost forever.

v3.1 advances Warden from a monitoring tool to an **active operations platform** with three complementary feature areas:

1. **Agent Orchestration Controls** — Start, stop, and restart agent sessions directly from the dashboard. The operator manages the agent lifecycle without touching the CLI. This closes the control loop: Warden already shows what agents are doing, now it lets the operator decide _whether_ they should be doing it.

2. **Enhanced Token Insights** — Burn rate calculation, budget alerts, model cost comparison, and cost projection. The operator moves from knowing _how much was spent_ to understanding _how fast money is being spent_ and _when a budget will be exceeded_. This enables proactive cost management instead of after-the-fact surprise.

3. **Session Recording & Replay** — Capture terminal output as lightweight timestamped recordings and replay them later at variable speed. This captures institutional knowledge (how did the agent solve that problem?), enables debugging (what happened in the 3am session?), and provides an audit trail for complex operations.

These three areas share a common theme: **the operator gains agency**. Control over agent lifecycle, visibility into cost velocity, and preservation of session history — each removes a reason to leave the Warden dashboard and reach for external tools.

All three areas build on existing infrastructure. Agent orchestration extends TmuxSessionManager and InstanceTracker. Token insights extend SessionUsageReader and the token_usage table. Session recording extends TerminalStreamService's PTY output pipeline. No architectural rewrites are required.

## Feature Area 1: Agent Orchestration Controls

_Start, stop, and restart agent sessions from the dashboard._

Requirement ID prefix: **ORCH-**

### Priority P1

**ORCH-01: Start a new agent session from the dashboard**

Launch a new Claude Code agent session by selecting an agent from the OpenClaw config, optionally specifying a project directory or working directory, and clicking Start. Warden spawns a tmux session with the standard naming convention (`{agentId}-{projectSlug}-{shortUuid}`) and starts Claude Code inside it.

- **Existing infrastructure:** GSD ControlsTab already has spawn dispatch capability via `POST /api/gsd/spawn` which calls `spawn.sh`. TmuxSessionManager has `createSessionForAgent()`. InstanceTracker auto-discovers new tmux sessions within 10 seconds.
- **Implementation:** New API endpoint `POST /api/instances/start` that accepts `{ agentId, projectPath?, workDir? }`. Server validates inputs (path prefix assertion per v2.1 security decision), calls `spawn.sh` or equivalent, returns 202 Accepted. InstanceTracker picks up the new session on next poll cycle.
- **Acceptance:** Operator selects agent, clicks Start, tmux session appears in tab bar within 15 seconds. Claude Code is running inside the new session.

**ORCH-02: Stop a running agent session gracefully**

Stop an active agent session with a two-phase shutdown: first send `Ctrl+C` to the Claude Code process inside the tmux session (giving it a chance to clean up), wait a grace period, then kill the tmux session.

- **Existing infrastructure:** TmuxSessionManager has `destroySession()` (kill-session) and `sendPromptToSession()` (send-keys). InstanceTracker marks disappeared sessions as stopped.
- **Implementation:** New API endpoint `POST /api/instances/:id/stop`. Server sends `tmux send-keys C-c` to the session, waits 5 seconds grace period, then calls `tmux kill-session`. InstanceTracker marks session as "stopping" immediately, then "stopped" after tmux session disappears.
- **Acceptance:** Operator clicks Stop on a running session. Confirmation dialog appears. After confirmation, session transitions to "stopping" state. Within 10 seconds, session shows as "stopped".

**ORCH-03: Restart a stopped or errored agent session**

Restart a session by stopping it (if still running) and then starting a new session with the same agent and project configuration.

- **Implementation:** Combines ORCH-02 (stop) and ORCH-01 (start) in sequence. `POST /api/instances/:id/restart`. If the session is still running, stop it first. Then start a new session with the same agentId and projectPath from the original session record.
- **Acceptance:** Operator clicks Restart on a session. If running, it stops first. New session appears in tab bar within 15 seconds with the same agent identity.

**ORCH-04: Agent session status lifecycle**

Extend the `AgentInstanceStatus` type to include transitional states that reflect the full session lifecycle. The tab bar and session list update in real time as sessions transition between states.

- **Current states:** `active`, `idle`, `stopped`, `error`
- **New states:** `starting` (session creation initiated, not yet discovered by InstanceTracker), `stopping` (stop command sent, awaiting tmux session termination)
- **Implementation:** Extend `AgentInstanceStatus` in `src/shared/types.ts`. InstanceTracker sets `starting` when start endpoint is called, transitions to `active` when tmux session appears. Sets `stopping` when stop endpoint is called, transitions to `stopped` when tmux session disappears.
- **Acceptance:** Session status badges in InstanceTabBar reflect lifecycle state (starting/active/stopping/stopped) in real time. Transitional states show spinner or pulse animation.

**ORCH-05: Safety guards (confirmation dialogs, double-start prevention)**

Prevent destructive or redundant operations. Stop and restart actions require explicit confirmation. Starting a new session for an agent that already has an active session is blocked or warned.

- **Implementation:** Client-side confirmation dialog (modal or inline) before stop/restart. Start button disabled or shows warning badge when agent already has an active session. Server-side guard returns 409 Conflict if attempting to start a session for an agent with an existing active session (unless force flag is set).
- **Acceptance:** Stop/restart show confirmation dialog. Start button disabled for agents with active sessions. Server rejects duplicate starts with 409.

## Feature Area 2: Enhanced Token Insights

_Burn rate, budget alerts, model comparison, cost projection, and data export._

Requirement ID prefix: **TOKN-**

### Priority P1

**TOKN-10: Burn rate calculation (tokens/hour, cost/hour over sliding window)**

Calculate and display the rate of token consumption and cost accumulation over configurable time windows. The operator sees not just totals but velocity — how fast money is being spent right now.

- **Existing infrastructure:** `SessionUsageReader` scans JSONL files every 5 minutes and populates the `token_usage` SQLite table with daily aggregates. `TokenUsageView` displays summary cards per agent.
- **Implementation:** New SQL queries against `token_usage` table with time-windowed aggregation. Compute cost per hour over last 1h, 4h, and 24h windows. Expose via `GET /api/token-usage/burn-rate?window=1h|4h|24h`. Client displays burn rate in TokenUsageView summary cards with a sliding window selector.
- **Note:** For sub-day granularity, the JSONL scanner may need to store hourly (not just daily) aggregates, or compute burn rate from raw JSONL timestamps on demand.
- **Acceptance:** TokenUsageView shows burn rate (cost/hour) for each agent with sliding window selector (1h/4h/24h). Burn rate updates on each scan cycle.

**TOKN-11: Budget alerts — per-agent daily cost threshold with visual warnings**

Set a daily cost threshold per agent. When an agent approaches or exceeds the threshold, Warden shows a visual warning. This prevents cost overruns before they happen.

- **Implementation:** New `budget_config` table in SQLite storing `{ agentId, dailyLimitUsd }`. Settings UI in TokenUsageView or a new Budget Settings panel to configure thresholds per agent. Warning badge at 80% of threshold (amber), alert badge at 100% (red). Badge visible on the History nav tab when any agent exceeds threshold.
- **Acceptance:** Budget threshold configuration stored per-agent. Visual warning at 80% and alert at 100%. Budget alerts visible as colored badge on the History nav tab when any agent exceeds threshold.

### Priority P2

**TOKN-12: Model comparison view — cost-per-task breakdown by model across agents**

Side-by-side comparison of token costs broken down by model variant (sonnet/opus/haiku). Helps the operator decide which model to assign to which agent for cost optimization.

- **Existing infrastructure:** `SessionUsageReader` already tracks model names via JSONL parsing. The `MODEL_PRICING` map has per-model rates. However, the current `token_usage` table aggregates across models — model-level breakdown requires schema extension or a new table.
- **Implementation:** Extend JSONL scanner to store per-model daily aggregates (new `token_usage_by_model` table or add `model` column to existing table). New `GET /api/token-usage/by-model` endpoint. Client displays bar chart or table comparing cost per model per agent.
- **Acceptance:** Model comparison view shows cost breakdown by model variant per agent as bar chart or table. Data is accurate per the JSONL source files.

**TOKN-13: Cost projection — estimated daily/weekly spend at current burn rate**

At the current burn rate (from TOKN-10), project forward to estimate daily and weekly total spend. Gives the operator a forward-looking view instead of only historical data.

- **Implementation:** Compute from burn rate data: `projectedDailyCost = burnRatePerHour * 24`. Display alongside burn rate in TokenUsageView. Weekly projection = daily * 7 (simple extrapolation with caveat label).
- **Acceptance:** Cost projection line shows estimated daily spend at current burn rate. Weekly projection available. Projection updates when burn rate window changes.

**TOKN-14: Export token usage data as CSV**

Download the full token usage dataset as a CSV file for external analysis in spreadsheets or BI tools.

- **Implementation:** New `GET /api/token-usage/export?format=csv` endpoint that queries the full `token_usage` table and returns a CSV response with headers: `date, agent_id, model, input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens, cost_usd`. Export button in TokenUsageView triggers download.
- **Acceptance:** Export button downloads CSV. CSV covers full date range in token_usage table. Columns match specification.

## Feature Area 3: Session Recording & Replay

_Capture terminal output streams and replay them later at variable speed._

Requirement ID prefix: **REC-**

### Priority P1

**REC-01: Capture terminal output stream to a recording file**

Record the raw PTY output of an active terminal session as timestamped chunks in a lightweight format. The recording captures exactly what appeared in the terminal, including ANSI escape sequences, cursor movement, and color codes — enabling faithful replay.

- **Existing infrastructure:** `TerminalStreamService` already receives all PTY output via the `onData` callback. The `setImmediate` tap pattern from Phase 11 demonstrates non-blocking output interception with zero latency impact.
- **Format:** asciicast v2 (JSON Lines with `[time, type, data]` tuples). This is the industry-standard format used by asciinema. Each line is a JSON array: `[elapsed_seconds, "o", "output_data"]`. Header line contains terminal dimensions and metadata.
- **Storage:** `data/recordings/` directory. One file per recording. SQLite metadata table (`recordings`) with columns: `id, sessionName, agentId, projectPath, startedAt, stoppedAt, durationSeconds, filePath, fileSizeBytes`.
- **Acceptance:** When recording is active for a session, PTY output is written to a recording file in asciicast v2 format. File grows in real-time. Recording metadata is stored in SQLite.

**REC-02: Recording toggle per session (start/stop from UI)**

The operator can start and stop recording for any active session from the terminal view header. Visual indicator shows when a session is being recorded.

- **Implementation:** Record button (circle icon) in terminal view header. Clicking starts recording — button turns red with pulse animation. Clicking again stops recording and finalizes the file. API endpoints: `POST /api/recordings/start` with `{ sessionName }`, `POST /api/recordings/stop` with `{ sessionName }`.
- **Acceptance:** Record button visible in terminal view header. Clicking starts recording (button turns red). Clicking again stops recording. Recording file is finalized and appears in recording library.

### Priority P2

**REC-03: Replay a recorded session at variable speed**

Open a completed recording in a read-only xterm.js terminal that replays the session output at the original timing, or at 2x/4x/8x speed. The operator can review what happened in a past session without the session being active.

- **Implementation:** New `RecordingPlayer` component with a read-only xterm.js terminal instance. Reads the asciicast v2 recording file via streaming API. Schedules output writes to xterm.js using the timestamp deltas from the recording. Speed controls adjust the delay multiplier.
- **Format parsing:** Read asciicast v2 lines, extract `[elapsed, "o", data]`, compute inter-event delays, apply speed multiplier.
- **Acceptance:** Clicking a recording opens a read-only xterm.js terminal. Output replays at original timing by default. Speed controls (1x, 2x, 4x, 8x) adjust playback speed. Playback can be paused and resumed.

**REC-04: Recording library — list past recordings with session metadata**

A browsable list of all past recordings showing agent name, project, date, duration, and file size. The operator can find and open any recording from this library.

- **Implementation:** New `RecordingLibrary` component in the History view. Queries `GET /api/recordings` endpoint which reads from the `recordings` SQLite table. List displays: agent name, project slug, recording date, duration, file size. Click to open in RecordingPlayer (REC-03).
- **Acceptance:** Recording library shows list of past recordings. Each entry shows agent, project, date, duration. Clicking opens the recording in the replay player.

### Priority P3

**REC-05: Auto-record option (configurable recording triggers)**

Instead of manually starting recordings, the operator can configure automatic recording rules. Options: always record all sessions, never auto-record, or record only when a permission prompt is detected (capturing the context around operator decisions).

- **Implementation:** Settings panel with auto-record dropdown: `always` | `never` | `on-permission-prompt`. Stored in SQLite `settings` table or `config` table. When set to `always`, TerminalStreamService automatically starts recording when a new PTY session is created. When set to `on-permission-prompt`, recording starts when the permission detection system (from Phase 19) fires.
- **Acceptance:** Settings panel offers auto-record options. When set to `always`, new sessions start recording automatically. When set to `on-permission-prompt`, recording starts when permission prompt is detected.

## Technical Notes

### Agent Start (ORCH-01)

Reuse the `spawn.sh` pattern established in v2.1 (Phase 12). The existing `POST /api/gsd/spawn` endpoint demonstrates the fire-and-forget pattern: POST returns 202 Accepted immediately while the tmux session starts in the background. `spawn.sh` blocks for 15-25 seconds; InstanceTracker discovers the new session within 10 seconds of tmux session creation.

Security: Apply the same path validation from v2.1 — `path.resolve()` + prefix assertion that result starts with `/home/forge/`. The `firstCommand` character allowlist `[/a-zA-Z0-9 @:._-]` prevents shell injection via tmux send-keys.

### Agent Stop (ORCH-02)

Two-phase graceful shutdown:
1. `tmux send-keys -t {sessionName}:0.0 C-c` — sends interrupt signal to Claude Code process
2. Wait 5-second grace period for clean shutdown
3. `tmux kill-session -t {sessionName}` — force-terminate the tmux session

InstanceTracker detects the disappeared session on next poll (within 10 seconds) and marks it as `stopped`. The transitional "stopping" state is set immediately when the stop endpoint is called and tracked in SQLite.

### Burn Rate Calculation (TOKN-10)

The current `token_usage` table stores daily aggregates. For hourly burn rate calculation with sub-day windows (1h, 4h), two approaches:

- **Option A:** Extend SessionUsageReader to store hourly aggregates in a new `token_usage_hourly` table. Higher storage cost but enables fast SQL queries.
- **Option B:** Compute burn rate on demand from raw JSONL timestamps. Lower storage but higher compute cost per request.

Recommended: Option A (hourly aggregates). The JSONL scanner already runs every 5 minutes — extending it to compute hourly buckets is straightforward. SQL queries like `SELECT SUM(cost_usd) / COUNT(DISTINCT hour) FROM token_usage_hourly WHERE timestamp > datetime('now', '-4 hours')` are efficient with an index on the timestamp column.

### Recording Format (REC-01)

**asciicast v2** — the industry-standard format for terminal session recording, used by asciinema. Format specification:

```
{"version": 2, "width": 120, "height": 40, "timestamp": 1709537485, "env": {"TERM": "xterm-256color"}}
[0.0, "o", "$ "]
[0.5, "o", "hello world\r\n"]
[1.2, "o", "$ "]
```

Line 1: Header with terminal dimensions and metadata (JSON object).
Subsequent lines: Event tuples as JSON arrays — `[elapsed_seconds, event_type, data]`.
- `"o"` = output (terminal data)
- `"i"` = input (optional, for recording operator keystrokes)

This format is:
- Human-readable (JSON Lines)
- Lightweight (text, not video — typical session ~1-5 MB/hour)
- Compatible with existing players (asciinema-player web component)
- Easy to generate from the PTY `onData` callback

### Recording Storage (REC-01)

- Directory: `data/recordings/`
- Filename pattern: `{sessionName}_{timestamp}.cast`
- SQLite metadata table: `recordings` with columns for session info, timestamps, duration, file path, file size
- Storage management: Consider rotation policy (delete recordings older than 30 days, or cap total storage at 10 GB) — defer to Phase 25 or later

### No New npm Dependencies (Phase 21-23)

Phases 21 through 23 require no new npm dependencies:
- Agent orchestration: TmuxSessionManager + InstanceTracker extensions, tmux CLI calls via execFile
- Token insights: SQL queries, simple arithmetic, CSV string generation
- Budget alerts: SQLite table, REST endpoints, React state management

### Phase 24 Dependency Consideration

Phase 24 (Recording & Replay) may benefit from the `asciinema-player` web component for replay, but a custom player using xterm.js directly is also viable and avoids adding a dependency. Decision deferred to Phase 24 planning.

## Out of Scope for v3.1

- **Multi-user access control or role-based permissions** — Warden is a single-operator tool (IP-whitelisted)
- **Agent creation or deletion from dashboard** — Agents are defined in `openclaw.json`, not managed by Warden
- **Remote server agent management** — Single-server deployment only; no SSH or remote tmux
- **Video recording of terminal** — Use lightweight asciicast text format, not screen capture
- **Real-time streaming replay** — Only completed recordings can be replayed; no live-follow mode
- **Mobile UI improvements** — Deferred from v2.0; xterm.js mobile touch fundamentally broken
- **Telegram alert forwarding** — Was P3 in v3.0, still deferred
- **detectAgentState() rewrite** — Regex heuristics remain fragile but functional; deferred
- **Multi-model budget tracking** — Budget alerts are per-agent total, not per-model (simplifies v3.1; extend in v3.2 if needed)
- **Recording sharing or export** — Recordings are local to the server; no download/upload/share features in v3.1

## Dependencies and Risks

### Dependencies

| Phase | Depends On | Nature |
|-------|-----------|--------|
| Phase 21 (Agent Orchestration) | Phase 20 infrastructure | TmuxSessionManager, InstanceTracker, spawn.sh pattern from v2.1 |
| Phase 22 (Burn Rate & Budget) | Phase 18 infrastructure | SessionUsageReader, token_usage table, TokenUsageView component |
| Phase 23 (Analytics & Export) | Phase 22 | Burn rate data structures, per-model aggregation |
| Phase 24 (Recording & Replay) | Phase 20 infrastructure | TerminalStreamService PTY output pipeline, setImmediate tap |
| Phase 25 (Recording Automation) | Phase 24 + Phase 19 | Recording infrastructure + permission detection system |

### Risks

1. **Agent start/stop process lifecycle management** — Error handling must be robust. A failed stop (tmux session won't die) must not leave the system in an inconsistent state. Mitigation: timeout-based fallback to force kill, InstanceTracker reconciliation on next poll.

2. **Claude Code session JSONL format stability** — The JSONL format is not a documented API and could change between Claude Code versions. SessionUsageReader already handles this gracefully (skip malformed lines, warn on unknown models). Risk is low but non-zero for v3.1 token features.

3. **Recording storage disk exhaustion** — On a single server, recordings could grow large (1-5 MB/hour per session, potentially dozens of concurrent sessions). Mitigation: implement storage limits in Phase 25 stretch, monitor disk usage, and document manual cleanup procedure.

4. **Sub-day token granularity** — Current token_usage table stores daily aggregates only. Burn rate calculation for 1h/4h windows requires either hourly aggregation (schema change) or on-demand JSONL parsing (performance concern). Decision needed at Phase 22 planning time.

5. **tmux CLI reliability for orchestration** — Agent start/stop depends on tmux responding correctly to CLI commands. tmux server crashes, zombie sessions, or permission issues could cause failures. Mitigation: InstanceTracker reconciliation provides eventual consistency; orchestration endpoints return clear error messages.

6. **asciicast format compatibility** — While asciicast v2 is an established standard, generating it correctly requires careful timestamp handling and proper escaping of terminal output in JSON. Mitigation: test with known asciinema player to verify format compliance.
