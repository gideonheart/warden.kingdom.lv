## Motivation

These two feature areas combine naturally because they both address the same operator pain point: _working effectively within the Warden dashboard without needing external tools or manual context-switching._

**Operator Awareness** is about passive monitoring — alerts come to the operator. The agent is blocked by a permission prompt or approaching context limits? Warden tells you without requiring you to stare at the terminal output.

**Terminal Power Tools** is about active investigation — the operator goes to the data. Need to find when an error occurred in 10,000 lines of scrollback? Ctrl+F. Need to switch sessions without reaching for the mouse? Keyboard shortcut.

Together, these features make Warden a **complete operator workstation**, not just a live terminal viewer:
- Passive: Warden watches agents and alerts you when they need attention
- Active: Warden gives you the tools to investigate and navigate efficiently when you do look

Both areas are additive (no architectural changes to existing services), use existing infrastructure (PTY tap, Socket.IO, xterm.js addon ecosystem), and deliver high operator value relative to implementation cost.

## Feature Area 1: Operator Awareness

### Priority P1 — Phase 19 candidates (ship first)

**1.1 Permission prompt detection with tab badge alert**

Detect when Claude Code is waiting for user permission in a terminal session. Claude Code prints a recognizable prompt when it needs operator approval before proceeding (e.g., tool use confirmation, file write, shell command). Detecting this early prevents silent agent blocking.

- **Detection:** Regex match on PTY output using the existing `setImmediate` tap mechanism from Phase 11 (`TerminalStreamService`). Patterns to match: `"Do you want to"`, `"Allow"`, `"Press Enter to continue"`, `"(y/N)"`, `"Overwrite?"`, plus Claude Code-specific permission prompt formats.
- **Signal path:** Backend detects pattern → emits `terminal:permission_prompt` event on Socket.IO `/terminal` namespace → frontend receives → sets badge state on the affected session tab.
- **Visual output:** Red badge dot on the `InstanceTabBar` tab for the affected session. Badge clears on next user input to that session (i.e., once the operator acknowledges the prompt).
- **Acceptance:** Badge appears within 5 seconds of permission prompt appearing in the PTY output. Badge clears on operator input.

**1.2 Context window pressure badge in terminal view**

The context pressure percentage is already parsed by `extractContextPressure()` in `useAgentLiveStatus.ts` (Phase 14). Currently only visible in the GSD Agents tab. This feature brings it into the main terminal view header where operators spend most of their time.

- **Signal path:** Context pressure is available via existing GSD live status polling. Add a `contextPressure` field to the Socket.IO session state or expose it via a lightweight REST endpoint polled from the terminal view header.
- **Visual output:** Percentage badge in the terminal view header area (alongside session name). Color thresholds: green below 70%, amber 70-89%, red 90%+. No badge when session has no context data.
- **Acceptance:** Context pressure badge visible in terminal view header. Updates when GSD data refreshes (every ~10s). Shows correct color for threshold.

### Priority P2 — Phase 20 candidates

**1.3 Agent state banner in terminal view**

The detected agent state (`idle` / `working` / `error` / `permission_prompt`) is available from `useAgentLiveStatus` but only shown in the GSD Agents tab. Surface it as a subtle status bar or overlay within the terminal view itself.

- **Implementation:** Small status chip in the terminal header bar (e.g., `[working]` in amber, `[permission_prompt]` in red, `[idle]` in muted gray). Does not overlay the terminal canvas — placed in the header/toolbar area.
- **Acceptance:** State chip visible in terminal view. Updates when agent state changes.

**1.4 Browser notification for permission prompts (opt-in)**

When a permission prompt is detected and the browser tab is not focused, fire a browser `Notification` API notification.

- **Implementation:** Request `Notification` permission once on first detection. Use `Notification` API only — no server-side component. Opt-in UX (user can dismiss permission request).
- **Acceptance:** Browser notification fires when permission prompt detected and browser tab is backgrounded. Does not fire when tab is focused (already visible).

## Feature Area 2: Terminal Power Tools

### Priority P1 — Phase 19 candidates (ship first)

**2.1 Terminal text search (Ctrl+F)**

Implement `xterm-addon-search` to find text in the xterm.js terminal buffer. Show a search bar overlay triggered by Ctrl+F or a toolbar button.

- **Package:** `xterm-addon-search` — the official xterm.js search addon. Verify npm compatibility with xterm.js 5.x before adding.
- **UI:** Search bar overlay at the top of the terminal canvas (not above the header). Contains: text input, "Previous" button, "Next" button, close button (or Escape to close). Non-blocking — operator can see terminal behind search bar.
- **Behavior:** Search as you type (no Enter required). Highlights matching text in the buffer. Pressing Enter or clicking Next advances to next match.
- **Acceptance:** Ctrl+F opens search overlay. Text entered highlights matches in terminal buffer. Next/Prev navigate between matches. Escape closes search. Works on full scrollback buffer, not just visible area.

**2.2 Keyboard navigation shortcuts**

Document and implement keyboard shortcuts for core Warden actions. These reduce mouse dependency for operators monitoring multiple sessions simultaneously.

- **Tab switching:** `Ctrl+1` through `Ctrl+9` switch to session tab by index. `Alt+Left` / `Alt+Right` cycle through tabs.
- **Terminal focus:** `Escape` (when search bar is closed) focuses the terminal canvas.
- **Open search:** `Ctrl+F` opens search overlay (handled by 2.1 above).
- **Toggle sidebar:** `Ctrl+B` (or similar) collapses/expands the `AgentSidebar`.
- **Implementation:** `keydown` listener on `document` in a `useHotkeys`-style hook or raw event listener. Use `event.preventDefault()` for captured shortcuts to prevent browser defaults. Scope shortcuts to avoid conflicting with terminal input (check `document.activeElement` or use focus guards).
- **Acceptance:** All shortcuts work in production build. Ctrl+1 through Ctrl+9 navigate to correct sessions. Tab cycling works. Shortcuts documented in a visible help tooltip or keyboard shortcut panel.

---

### Priority P2 — Phase 20 candidates

**2.3 Search match count and highlight persistence**

Enhance the search overlay from 2.1 to show match count and maintain visible highlights while navigating.

- **Visual output:** Counter in search bar showing current position and total matches (e.g., `3 of 17 matches`). Updates on each keypress. Shows "No matches" when search text finds nothing.
- **Highlight persistence:** Search highlights remain visible when navigating between matches (xterm-addon-search default behavior — verify).
- **Acceptance:** Match count displayed correctly. Highlights persist across Next/Prev navigation.

**2.4 Scrollbar gutter markers**

Visual markers in a custom scrollbar gutter (overlaid on the terminal) indicating where search matches or permission prompt lines appear in the buffer.

- **Implementation:** Canvas overlay or CSS overlay alongside the terminal. Markers at proportional positions representing match locations in the full buffer.
- **Complexity:** Medium-high. May require custom scrollbar implementation. Defer if too complex in Phase 20.
- **Acceptance:** Gutter markers appear for search results. Clicking a marker scrolls terminal to that position.

---

### Priority P3 — Phase 21 / stretch

**2.5 Terminal bookmarks**

Let operators mark positions in the scrollback buffer for later reference.

- **Implementation:** `Ctrl+M` (or button) sets bookmark at current scroll position. Bookmarks listed in a panel or accessible via keyboard shortcut.

**2.6 Regex search**

Support regex patterns in the terminal search overlay.

- **Implementation:** Toggle button in search bar to switch between literal and regex mode. `xterm-addon-search` supports regex via `regex: true` option in `SearchDecorationOptions`.

## Technical Notes

### xterm-addon-search

- Package: `xterm-addon-search` on npm
- Must verify compatibility with xterm.js `^5.x` before adding as dependency
- Provides `SearchAddon` class with `findNext(term, options?)` and `findPrevious(term, options?)`
- Search options include `regex`, `wholeWord`, `caseSensitive`, `incremental`, `decorations`
- No new architectural dependencies — pure xterm.js ecosystem addition

### Permission prompt detection

- Reuse the PTY output tap from Phase 11 (`TerminalStreamService`, `setImmediate` pattern)
- Regex patterns to cover: `Do you want to proceed`, `Allow`, `Press Enter to continue`, `\(y\/N\)`, `Overwrite\?`, and Claude Code permission prompt format (inspect actual output to confirm)
- Emit `terminal:permission_prompt` event on the `/terminal` Socket.IO namespace
- Backend tracks per-session state: `{ sessionId: string, hasPermissionPrompt: boolean }`
- Clear flag when `terminal:input` received (operator acknowledges prompt)

### Context pressure surfacing

- `extractContextPressure()` already parses the percentage from tmux pane output (Phase 14, `useAgentLiveStatus.ts`)
- Current availability: GSD Agents tab via `useAgentLiveStatus` polling hook
- Options for surfacing in terminal view:
  - Option A: Add context pressure to the Socket.IO session state (`TerminalStreamService`)
  - Option B: Lightweight REST endpoint `GET /api/instances/:id/status` polled from terminal view header every 10s
  - Option B is lower coupling — terminal view stays decoupled from GSD plugin infrastructure
- Prefer Option B for Phase 19; revisit in Phase 20 if polling frequency is a concern

### Keyboard shortcuts

- Pattern: `keydown` listener on `document` with `event.preventDefault()` for captured shortcuts
- Focus guard: check `event.target` — do NOT intercept shortcuts when focus is inside a text input or textarea (other than terminal canvas)
- Terminal canvas (`xterm.js`) captures keypresses internally — shortcuts need to be registered at document level to work when terminal is focused
- Consider `useHotkeys` library pattern or simple raw listener — no new dependencies preferred (raw listener is straightforward for ~6 shortcuts)

### No new npm dependencies beyond xterm-addon-search

All planned features map to existing infrastructure:
- PTY tap: Phase 11 mechanism, already in TerminalStreamService
- Socket.IO events: existing `/terminal` namespace, just new event types
- Context pressure: already computed by extractContextPressure()
- Browser Notification API: web standard, no library needed
- Keyboard shortcuts: raw DOM event listener, no library needed
- xterm-addon-search: the one new dependency (official xterm.js ecosystem)

## Milestone Acceptance Criteria

1. Permission prompts are visually flagged within 5 seconds of appearing in any active session
2. Context pressure is visible in the terminal view header (not just GSD tab) when above threshold
3. Ctrl+F opens terminal search and finds text in the full scrollback buffer
4. Keyboard shortcuts exist for tab switching (Ctrl+1-9), tab cycling (Alt+Left/Right), and sidebar toggle (Ctrl+B)
5. All features work in production build (`npm run build && npm start`)
6. Playwright E2E tests cover search overlay open/close and keyboard shortcut tab navigation
7. No regressions in existing E2E tests

---

## Out of Scope for This Milestone

- Mobile terminal improvements (deferred — xterm.js mobile touch fundamentally broken, ~5 year issue)
- Multi-pane terminal layouts
- Agent creation or lifecycle management
- Real-time collaboration features (multi-operator)
- Terminal recording or session replay
- Any changes to the Gateway integration or OpenClaw config format
