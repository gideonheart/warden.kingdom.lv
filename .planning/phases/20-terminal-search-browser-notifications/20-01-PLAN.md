**Step 2: Create TerminalSearchOverlay component**

Create `src/client/components/TerminalSearchOverlay.tsx` — a floating search bar positioned top-right inside the terminal pane's `relative` wrapper.

Props interface:
```typescript
interface TerminalSearchOverlayProps {
  searchAddonRef: React.RefObject<SearchAddon | null>;
  onClose: () => void;
  initialQuery: string;
  onQueryChange: (query: string) => void;
  terminalFocusRef?: React.MutableRefObject<(() => void) | null>;
}
```

Implementation details:
- **Layout**: Absolutely positioned `top-2 right-2` inside the terminal content area. Uses `z-40` (above terminal, below connecting overlay). Background `bg-warden-panel` with border and rounded corners. Does NOT push terminal content down — floats above it.
- **Animation**: Use CSS transition classes. The component receives an `isVisible` boolean-like control from the parent, but here we render conditionally in TerminalView. On mount, use a `useEffect` that sets a `mounted` state to `true` after a tick to trigger the CSS transition from `opacity-0 translate-x-4` to `opacity-100 translate-x-0`. Duration ~150ms ease-out.
- **Input**: Auto-focus the search `<input>` on mount via `useEffect` + `inputRef.current?.focus()`. The input receives `initialQuery` as default value so the query persists across close/reopen within the same session.
- **Debounced search**: `useEffect` on `query` state with 300ms `setTimeout`. On fire, calls `searchAddonRef.current?.findNext(query, { decorations: ... })`. Cleanup clears the timeout. Use `useRef<ReturnType<typeof setTimeout> | null>` for the timer ref, with cleanup in the useEffect return AND a separate unmount useEffect.
- **Decoration colors** (locked by user):
  - `matchBackground: '#f59e0b33'` (yellow, semi-transparent)
  - `matchBorder: '#f59e0b'` (solid yellow)
  - `matchOverviewRuler: '#f59e0b'` (gutter tick = yellow)
  - `activeMatchBackground: '#f97316'` (orange)
  - `activeMatchBorder: '#f97316'`
  - `activeMatchColorOverviewRuler: '#f97316'`
- **Match count display**: Subscribe to `searchAddon.onDidChangeResults(({ resultIndex, resultCount }) => {...})` in a `useEffect`. Guard: `if (!addon) return;`. Display format:
  - No query or no matches: "No results"
  - `resultCount >= 1000`: `"${resultIndex + 1} / 1000+"`
  - Normal: `"${resultIndex + 1} / ${resultCount}"`
  - When `resultIndex === -1`, show "0 / {count}" or "No results" depending on count.
- **Navigation**: Two buttons (Previous / Next) styled as small `bg-warden-border/40` buttons with chevron characters. Also handle keyboard:
  - `Enter` → `searchAddonRef.current?.findNext(query, { decorations: ... })`
  - `Shift+Enter` → `searchAddonRef.current?.findPrevious(query, { decorations: ... })`
  - `Escape` → `event.preventDefault(); event.stopPropagation(); onClose();` then `requestAnimationFrame(() => terminalFocusRef?.current?.());` — the stopPropagation prevents useGlobalHotkeys from also handling Escape.
- **Clear on empty**: When query becomes empty string, call `searchAddonRef.current?.clearDecorations()` and reset match count to 0.
- **onQueryChange callback**: Call `onQueryChange(query)` whenever the query changes, so TerminalView can persist the query for reopen.
- **Cleanup**: On unmount, call `searchAddonRef.current?.clearDecorations()` to remove highlights when overlay closes.

**Step 3: Handle Escape interaction with search**

The existing Escape handler in useGlobalHotkeys refocuses the terminal. When search overlay is open, Escape should close the overlay (handled by the overlay's own onKeyDown). The overlay calls `event.stopPropagation()` on Escape, which prevents useGlobalHotkeys from also responding. No changes needed in useGlobalHotkeys for Escape — the overlay's stopPropagation is sufficient since it's a React-level handler that fires before the document-level capture handler... wait, actually document capture fires FIRST. Let me reconsider.

The useGlobalHotkeys capture-phase handler fires first (document level). But it has the `isInTextInput` guard — when the search input is focused, `event.target` is an INPUT element, so the entire handler returns early. This means Escape presses while the search input is focused will NOT be handled by useGlobalHotkeys, and will naturally reach the overlay's own onKeyDown handler. This is correct behavior — no changes needed.

But what about when the user presses Ctrl+F while search is already open? It should toggle it off (close) or at minimum re-focus the search input. Add logic in TerminalView's searchOpenRef callback:
```typescript
searchOpenRef.current = () => {
  setIsSearchOpen((prev) => {
    if (prev) {
      // Already open — just re-focus the input
      // The overlay auto-focuses on mount, but we need to re-focus
      // This is handled by the overlay itself on render
      return prev;
    }
    return true;
  });
};
```
Actually simpler: just always set to true. The overlay auto-focuses on mount. If already open, the input already has focus. No toggle behavior needed for Ctrl+F — it's "open search", not "toggle search".
