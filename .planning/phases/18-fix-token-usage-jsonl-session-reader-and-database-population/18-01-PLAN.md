<objective>
Create the SessionUsageReader service that scans Claude Code JSONL session files, extracts token usage data from assistant messages, and populates the token_usage SQLite table with aggregated daily usage per project directory.

Purpose: The token_usage table has existed since v1.0 but has never been populated. Claude Code stores detailed per-API-call usage data in JSONL session files under ~/.claude/projects/. This plan creates the backend service to read those files, aggregate usage by date, and store results in SQLite so the existing TokenUsageView UI can display real data.

Output: Working SessionUsageReader service, updated DB schema with cache token columns, and upsertTokenUsage DB method.
</objective>

<task type="auto">
  <name>Task 2: Create SessionUsageReader service</name>
  <files>src/server/services/SessionUsageReader.ts</files>
  <action>
Create a new service `SessionUsageReader` at src/server/services/SessionUsageReader.ts that:

1. **Constants:**
   - `CLAUDE_PROJECTS_DIR = path.resolve(process.env.HOME ?? '/home/forge', '.claude/projects')`
   - `SCAN_INTERVAL_MS = 5 * 60 * 1000` (5 minutes)
   - Model pricing map (per 1M tokens): `{ 'claude-opus-4-6': { input: 15, output: 75, cacheWrite: 18.75, cacheRead: 1.5 }, 'claude-sonnet-4-6': { input: 3, output: 15, cacheWrite: 3.75, cacheRead: 0.30 }, 'claude-haiku-4-5-20251001': { input: 0.80, output: 4, cacheWrite: 1, cacheRead: 0.08 } }` with a fallback to claude-sonnet-4-6 pricing for unknown models.

2. **Core method: `scanAllProjects(): Promise<void>`**
   - Read all directories under CLAUDE_PROJECTS_DIR using readdir
   - For each project directory:
     a. Derive a project label from the directory name (replace leading `-` with `/`, then all `-` with `/` — but this produces ambiguity, so just use the raw directory name as the `agent_id` for now, stripped of the leading dash, e.g. `home-forge-warden-kingdom-lv`)
     b. Find all `*.jsonl` files (top-level and `*/subagents/*.jsonl`) using glob or recursive readdir
     c. For each JSONL file, parse all lines where `type === 'assistant'` and `message.usage` exists
     d. Extract: `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`, `model` from each assistant message
     e. Group usage by date (derive date from `timestamp` field, format as `YYYY-MM-DD`)
     f. Compute `cost_usd` using the model pricing map: `(input * inputPrice + output * outputPrice + cacheCreate * cacheWritePrice + cacheRead * cacheReadPrice) / 1_000_000`
   - For each (project_label, date) aggregate, call `database.upsertTokenUsage()`

3. **Performance considerations:**
   - Use `fs.open` + `fs.read` to read only the last portion of large files if needed? No — for accuracy, read entire files. Files are typically <2MB. Use `readFile` with `utf-8` encoding.
   - Skip lines that fail JSON.parse (partial writes, corrupted lines)
   - Catch and log errors per-file, don't let one bad file stop the scan

4. **Lifecycle methods:**
   - `startPeriodicScan(): void` — runs scanAllProjects immediately then on SCAN_INTERVAL_MS
   - `stopPeriodicScan(): void` — clears the interval
   - `scanAllProjects(): Promise<void>` — public for manual trigger

5. **Export:** `export const sessionUsageReader = new SessionUsageReader();`

Follow the same singleton pattern as InstanceTracker and other services.
