1. Create `src/server/services/RecordingRotationService.ts`:

```typescript
import fs from 'fs';
import { database } from '../database/DatabaseConnection.js';
import { recordingCaptureService } from './RecordingCaptureService.js';
```

Export class `RecordingRotationService` with:

- `private intervalHandle: ReturnType<typeof setInterval> | null = null`
- `private readonly CHECK_INTERVAL_MS = 5 * 60 * 1000` (5 minutes)

Methods:
- `startPeriodicRotation(): void` — Run `this.runRotation()` immediately on startup (cleanup any orphan `deletion_pending` rows from crashes), then `setInterval` at CHECK_INTERVAL_MS. Same pattern as `InstanceTracker.startPeriodicSync()`.
- `stopPeriodicRotation(): void` — clearInterval if handle exists, set to null.
- `runRotation(): { deletedCount: number; freedBytes: number }` — Synchronous (all DB/fs operations are sync with better-sqlite3):
  a. Get rotation config. If capBytes === 0, return { deletedCount: 0, freedBytes: 0 }.
  b. First, cleanup any existing deletion_pending rows from previous runs: query rows WHERE deletion_pending = 1, for each attempt fs.unlinkSync on filePath (ignore ENOENT), then database.deleteRecording(id). This handles the orphan cleanup on startup or after crashes.
  c. Get storage stats. If totalBytes <= capBytes, return early.
  d. Calculate overage = totalBytes - capBytes.
  e. Get rotation candidates (oldest completed, not pending, not null file_size).
  f. Loop through candidates: skip if `recordingCaptureService.isRecording(candidate.sessionName)`. Call `database.markDeletionPending(candidate.id)`. Then immediately attempt file deletion (fs.existsSync + fs.unlinkSync, catch errors). Then `database.deleteRecording(candidate.id)`. Add to freedBytes and deletedCount. Break when freedBytes >= overage.
  g. Return { deletedCount, freedBytes }.

Export a singleton: `export const recordingRotationService = new RecordingRotationService();`

Add 4 new endpoints AFTER the existing auto-record-config routes but BEFORE any `/:id` routes (before the `POST /api/recordings/session/:sessionName/start` route). This follows the same literal-before-param ordering lesson from Phase 30.

```typescript
// GET /api/recordings/storage-stats — current usage + cap
recordingRoutes.get('/api/recordings/storage-stats', (_req, res) => {
  const stats = database.getStorageStats();
  const config = database.getRotationConfig();
  res.json({ ...stats, capBytes: config.capBytes });
});

// GET /api/recordings/rotation-config — just the cap
recordingRoutes.get('/api/recordings/rotation-config', (_req, res) => {
  const config = database.getRotationConfig();
  res.json(config);
});

// PUT /api/recordings/rotation-config — set cap in bytes
recordingRoutes.put('/api/recordings/rotation-config', (req, res) => {
  const { capBytes } = req.body as { capBytes?: number };
  if (typeof capBytes !== 'number' || capBytes < 0) {
    res.status(400).json({ error: 'capBytes (non-negative number) is required' });
    return;
  }
  database.setRotationConfig(capBytes);
  res.json({ capBytes });
});

// POST /api/recordings/rotation/prune — manual prune trigger
recordingRoutes.post('/api/recordings/rotation/prune', (_req, res) => {
  const result = recordingRotationService.runRotation();
  res.json(result);
});
```
