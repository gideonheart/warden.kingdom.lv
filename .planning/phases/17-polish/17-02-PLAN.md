    **gsdRoutes.ts — Spawn handler fd safety (FIX-01):**
    In the POST /api/gsd/spawn handler (around lines 216-223), the current code is:
    ```
    const logFd = openSync(spawnLogPath, 'w');
    const child = spawn(SPAWN_SH_PATH, [...], { detached: true, stdio: ['ignore', logFd, logFd] });
    child.unref();
    ```

    If `spawn()` throws (e.g., SPAWN_SH_PATH missing), `closeSync` never runs and the fd leaks. Wrap in try/finally:
    ```
    const logFd = openSync(spawnLogPath, 'w');
    try {
      const child = spawn(SPAWN_SH_PATH, [...], { detached: true, stdio: ['ignore', logFd, logFd] });
      child.unref();
    } finally {
      closeSync(logFd);
    }
    ```

    **gsdRoutes.ts — extractContextPressure regex (FIX-04):**
    Current regex: `/(\d{1,3})%/` matches ANY percentage in the last 5 non-empty lines. This causes false positives when terminal output contains percentages (e.g., "npm install 45% complete", "test coverage 87%").

    The Claude Code status bar format shows context usage like:
    - `> 45% context used` or similar status line patterns
    - The percentage appears at the end of a status/progress bar line in the bottom of the terminal

    Replace the regex with one anchored to Claude Code's status bar format. The Claude Code TUI shows a status bar at the bottom with a pattern like colored blocks followed by a percentage. Look for lines where the percentage is preceded by block characters (Unicode block elements like `\u2588`, `\u2593`, `\u2591`) or a bar-like pattern, OR a line that contains both a percentage and typical status bar indicators.

    A practical anchor: Claude Code's status bar shows something like `[####    ] 45%` or colored block chars followed by percentage. The safest anchor is to look for a line where the percentage appears with block/bar characters OR is on a very short line (status bars are typically short). Use this pattern:
    ```
    /(?:[\u2580-\u259F\u2500-\u257F\u2800-\u28FF#=|].*?|.*context.*?)(\d{1,3})%/i
    ```
    This matches percentage preceded by Unicode box-drawing/block characters, or on a line containing "context". This is much more selective than the bare `(\d{1,3})%`.

    Alternatively, a simpler and more reliable approach: look for the percentage on a line that is SHORT (< 80 chars, typical for status bars) AND appears at the end of the last non-empty lines. Claude Code status bars are typically under 60 characters. Filter `lastFiveLines` to only consider lines shorter than 80 characters before applying the regex:
    ```typescript
    const statusCandidates = nonEmptyLines.slice(-5).filter((line) => line.trim().length < 80);
    const candidateText = statusCandidates.join('\n');
    const match = /(?:[\u2580-\u259F]|context).*?(\d{1,3})%/i.exec(candidateText);
    ```

    **Approach — useRef + JSON comparison:**

    2. Add a ref to track the serialized previous state: `const previousDataRef = useRef<string>('')`.

       ```typescript
       const serialized = JSON.stringify(Array.from(nextMap.entries()));

       setStatusMap(nextMap);
       ```

    This ensures `setStatusMap` only fires when agent data actually changes, keeping the Map reference stable across identical poll responses. The JSON.stringify cost is negligible for the small agent list (typically 3-8 agents).
