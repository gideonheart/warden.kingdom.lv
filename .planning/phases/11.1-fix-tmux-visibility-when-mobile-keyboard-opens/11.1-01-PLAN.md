<objective>
Fix terminal visibility and prompt sheet positioning when mobile virtual keyboard opens/closes.

Purpose: On iOS Safari, opening the keyboard fires `visualViewport.resize` not `window.resize`. The xterm.js FitAddon only listens to `window.resize`, so the terminal never refits and content is hidden behind the keyboard. Additionally, MobilePromptSheet uses `fixed bottom-0` which anchors to the layout viewport on iOS — sitting behind the keyboard.

Output: Two modified files with visualViewport listeners that make terminal content visible and prompt sheet accessible when keyboard is open on mobile devices.
</objective>

Add a new `useEffect` to the `MobilePromptSheet` component that dynamically sets `bottom` style based on the visual viewport offset when the keyboard opens:

1. Add a new `useEffect` after the existing click-outside effect (after line 40):
   ```typescript
   // Float above iOS keyboard by tracking visual viewport offset
   useEffect(() => {
     const viewport = window.visualViewport;
     const sheet = sheetRef.current;
     if (!viewport || !sheet) return;

     const updateBottomPosition = () => {
       const offsetFromLayoutBottom =
         window.innerHeight - (viewport.offsetTop + viewport.height);
       sheet.style.bottom = `${Math.max(0, offsetFromLayoutBottom)}px`;
     };

     viewport.addEventListener('resize', updateBottomPosition);
     viewport.addEventListener('scroll', updateBottomPosition);
     updateBottomPosition();

     return () => {
       viewport.removeEventListener('resize', updateBottomPosition);
       viewport.removeEventListener('scroll', updateBottomPosition);
       sheet.style.bottom = '';
     };
   }, []);
   ```

Key constraints:
- The empty dependency array `[]` is correct — `window.visualViewport` is a stable reference
- `Math.max(0, offset)` prevents negative bottom values when visual viewport is larger than expected
- Cleanup resets `sheet.style.bottom` to empty string so CSS `bottom-0` takes over again
- Listen to BOTH `resize` and `scroll` events on visualViewport — iOS can fire scroll events when keyboard animates
- Keep existing `fixed bottom-0` CSS class — when keyboard is closed, offsetFromLayoutBottom will be 0, so `style.bottom = '0px'` matches the CSS
- Do NOT change MobileKeyToolbar (it uses `sticky` which is already correct)
- Keep the existing `paddingBottom: 'env(safe-area-inset-bottom, 0px)'` inline style — it handles notch devices
