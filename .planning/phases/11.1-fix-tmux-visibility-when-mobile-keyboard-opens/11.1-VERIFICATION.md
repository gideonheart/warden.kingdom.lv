### Human Verification Required

#### 1. MobilePromptSheet floating above iOS keyboard

**Test:** On an iPhone (iOS 15+) using Safari, open the Warden dashboard terminal view. Tap the MobilePromptSheet collapsed bar to expand it. If a keyboard-triggering input appears, observe the sheet position.
**Expected:** The sheet is visible above the keyboard, not hidden beneath it. `sheet.style.bottom` should reflect a non-zero value equal to the keyboard height.
**Why human:** The `visualViewport.resize` / `scroll` events only fire from the iOS Safari engine when a real keyboard opens. Cannot simulate in grep-based or build-time checks.

#### 2. Terminal content remains visible when iOS keyboard opens

**Test:** On an iPhone (iOS 15+) using Safari, open a tmux session. Focus any input that triggers the keyboard (e.g., the paste textarea in MobileKeyToolbar). Observe the xterm.js terminal.
**Expected:** Terminal content reflows — the number of visible rows decreases to fit the visible viewport above the keyboard. Content is not clipped or hidden behind the keyboard.
**Why human:** The refit depends on `fitAddon.fit()` being called with correct container dimensions AFTER the CSS transition completes. Requires a real browser rendering context.

#### 3. Keyboard dismiss restores full terminal + prompt sheet positions

**Test:** After performing tests 1 and 2, dismiss the keyboard (tap outside the input or press Done).
**Expected:** Terminal expands to fill full visible height; MobilePromptSheet returns to `bottom: 0` (CSS class takes over after `style.bottom` is cleared on cleanup — but cleanup only fires on unmount, not keyboard dismiss).
**Why human:** The `updateBottomPosition` effect does not unmount on keyboard close — the offset will recalculate to 0 via the `visualViewport.resize` event. This is the intended behavior but requires live verification that the offset actually reaches 0 on close.

#### 4. No regression on Android Chrome

**Test:** On Android Chrome, open a terminal session and verify the keyboard open/close still triggers `window.resize` and refits the terminal as before.
**Expected:** Same behavior as pre-phase on Android — terminal refits, no visual regression.
**Why human:** Requires Android device or emulator.

### Gaps Summary

No automated gaps found. All four observable truths are fully implemented and wired:

- `refitTerminal` replaces the old `handleWindowResize` and listens to both `window.resize` (desktop/Android) and `visualViewport.resize` (iOS Safari). The 100ms debounce with `requestAnimationFrame` follows the plan exactly.
- The `terminal.onResize` → `sendResize` chain was already wired from a prior phase and correctly fires after `fitAddon.fit()`.
- `updateBottomPosition` in MobilePromptSheet correctly calculates keyboard height as `window.innerHeight - (viewport.offsetTop + viewport.height)` and applies it as `style.bottom`.
- Both effects clean up listeners on unmount.
- TypeScript type check passes with zero errors (confirmed via `npm run typecheck`).
- Both task commits (`2066260`, `b6e0de0`) exist in git history with appropriate commit messages and single-file scope.

The `human_needed` status reflects that the fix targets iOS Safari-specific browser behavior that cannot be verified without a real device — not any code-level deficiency.
