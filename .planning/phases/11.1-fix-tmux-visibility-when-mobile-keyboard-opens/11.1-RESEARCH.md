# Phase 11.1: Fix tmux Visibility When Mobile Keyboard Opens - Research

**Researched:** 2026-02-17
**Domain:** Mobile browser viewport behavior, xterm.js FitAddon, iOS/Android virtual keyboard CSS/JS
**Confidence:** HIGH

## Summary

When the mobile virtual keyboard opens in a browser, it does NOT simply "push content up" — it shrinks the **visual viewport** while leaving the **layout viewport** unchanged. The existing `.app-height` class already addresses this correctly by setting `height: var(--visual-viewport-height, 100dvh)`, which is updated via a `visualViewport` resize listener in App.tsx. However, there are two remaining gaps:

1. **The xterm.js FitAddon is not refitting on visual viewport resize.** TerminalView.tsx listens only to `window.resize`, which fires reliably on Android but is unreliable on iOS Safari when the keyboard opens. The `visualViewport` resize event is the reliable cross-platform signal. When the keyboard opens and `.app-height` shrinks, the xterm terminal canvas still occupies the old, larger dimensions — it becomes partially or fully hidden behind the keyboard because xterm.js has not recalculated rows and called `sendResize`.

2. **MobilePromptSheet uses `position: fixed`**, which anchors to the layout viewport on iOS. When the keyboard opens on iOS, `fixed bottom-0` elements remain at the layout viewport's bottom, which is beneath the keyboard — they do not automatically float above it without JavaScript intervention.

**Primary recommendation:** Add a `visualViewport` resize listener in TerminalView.tsx that calls `fitAddon.fit()` and `sendResize(cols, rows)` (with a small debounce), plus address the MobilePromptSheet positioning for keyboard-open state. The `index.html` already has `interactive-widget=resizes-content` which helps Chrome/Firefox but not iOS Safari — so the JS approach is required for full cross-platform coverage.

### Root cause diagnosis

**Terminal content invisible:**
The `.app-height` div correctly shrinks via `--visual-viewport-height`, but the xterm.js canvas inside it still has the old rendered dimensions. FitAddon calculates rows/cols based on container pixel size at the time `fit()` is called. Since `fit()` is only called on `window.resize`, and iOS Safari does not fire `window.resize` when the keyboard opens (it fires `visualViewport.resize` instead), the terminal never refits. The terminal renders at, say, 50 rows, but the visible container is now only tall enough for 30 rows — the bottom 20 rows are hidden behind the keyboard.

**MobilePromptSheet hidden:**
`fixed bottom-0` elements are positioned relative to the layout viewport, not the visual viewport. iOS Safari does not shrink the layout viewport when the keyboard opens. Therefore the prompt sheet remains at the old `bottom: 0` — underneath the keyboard. Android Chrome/Firefox are less affected because `interactive-widget=resizes-content` causes the layout viewport to resize.

## Architecture Patterns

### Pattern 1: Cross-Platform Visual Viewport Resize Handler in TerminalView

**What:** Add a `visualViewport` resize listener in TerminalView's `useEffect` that debounces and calls `fitAddon.fit()` then `sendResize()`.

**When to use:** Required for iOS Safari keyboard visibility fix. Complements existing `window.resize` handler (keep both — `window` covers desktop/Firefox/Chrome; `visualViewport` covers iOS Safari).

**Current code (incomplete):**
```typescript
// TerminalView.tsx L394-401 — EXISTING (keeps)
const handleWindowResize = () => {
  try {
    fitAddon.fit();
  } catch {
    // Container may have zero dimensions during layout transitions
  }
};
window.addEventListener('resize', handleWindowResize);
```

**Required addition:**
```typescript
// Add to the same useEffect in TerminalView.tsx
let resizeDebounceTimer: ReturnType<typeof setTimeout> | null = null;

const handleVisualViewportResize = () => {
  if (resizeDebounceTimer) clearTimeout(resizeDebounceTimer);
  resizeDebounceTimer = setTimeout(() => {
    requestAnimationFrame(() => {
      try {
        fitAddon.fit();
        // fitAddon.fit() triggers terminal.onResize which calls sendResize automatically
      } catch {
        // Container may have zero dimensions during transitions
      }
    });
  }, 100); // 100ms debounce — keyboard animation is ~250ms
};

const visualViewport = window.visualViewport;
if (visualViewport) {
  visualViewport.addEventListener('resize', handleVisualViewportResize);
}

// In cleanup:
return () => {
  if (resizeDebounceTimer) clearTimeout(resizeDebounceTimer);
  if (visualViewport) {
    visualViewport.removeEventListener('resize', handleVisualViewportResize);
  }
  window.removeEventListener('resize', handleWindowResize);
  // ...
};
```

**Key insight:** `terminal.onResize` fires automatically when `fitAddon.fit()` recalculates dimensions. The `onResize` callback in TerminalView already calls `sendResize(cols, rows)` — so no additional wiring is needed.

### Pattern 2: MobilePromptSheet Positioning Fix

**What:** The `MobilePromptSheet` uses `fixed bottom-0` which anchors to the layout viewport on iOS. Two approaches:

**Option A — CSS Visual Viewport positioning (simpler, preferred):**
Since `.app-height` already tracks `--visual-viewport-height`, and MobilePromptSheet is rendered as a sibling to the main `<div class="flex flex-col app-height">`, consider moving it INSIDE the app-height container as a normal flow `sticky` element, or use the `--visual-viewport-height` variable:

```tsx
// Instead of fixed bottom-0, use visual-viewport-aware bottom
<div
  className="fixed bottom-0 left-0 right-0 z-40 lg:hidden ..."
  style={{
    // On iOS: position relative to visual viewport bottom
    bottom: 'env(safe-area-inset-bottom, 0px)',
  }}
>
```
This alone won't fix it — the layout viewport bottom IS still below the keyboard.

**Option B — JS `visualViewport` offset (reliable, used by MDN example):**
```tsx
// In MobilePromptSheet, add effect:
useEffect(() => {
  const viewport = window.visualViewport;
  if (!viewport) return;

  const sheetEl = sheetRef.current;
  if (!sheetEl) return;

  const updatePosition = () => {
    // Position sheet at bottom of VISUAL viewport, not layout viewport
    const offsetBottom = window.innerHeight - viewport.height - viewport.offsetTop;
    sheetEl.style.bottom = `${offsetBottom}px`;
  };

  viewport.addEventListener('resize', updatePosition);
  viewport.addEventListener('scroll', updatePosition);
  updatePosition();

  return () => {
    viewport.removeEventListener('resize', updatePosition);
    viewport.removeEventListener('scroll', updatePosition);
  };
}, []);
```

**Option C — Move inside flow (cleanest long-term):**
Restructure so MobilePromptSheet is a flex child of the `app-height` div (inside the column, not overlaid), making it part of the natural flow. This avoids all fixed positioning issues. Downside: more DOM restructuring.

**Recommendation:** Option B for MobilePromptSheet (minimal refactor, reliable cross-platform). Option A for the main terminal area fix.

### Pattern 3: MobileKeyToolbar Already Correct

The `MobileKeyToolbar` uses `sticky bottom-0` inside a `flex-col` TerminalView container. This is correct — sticky elements are positioned within their scroll container, not the viewport. When `.app-height` shrinks, the flex column shrinks, and the sticky toolbar naturally rises. No changes needed here.

### Anti-Patterns to Avoid
- **Adding only `window.resize` listener:** Does not fire on iOS keyboard open.
- **Using VirtualKeyboard API (`navigator.virtualKeyboard`):** No iOS Safari support.
- **Removing `interactive-widget=resizes-content` from index.html:** This already helps Chrome/Firefox — keep it.
- **Calling `fitAddon.fit()` immediately in the resize handler:** Container dimensions may not have settled yet after CSS transitions. Always defer to `requestAnimationFrame` or a small setTimeout.
- **Calling `fit()` when container has zero dimensions:** Already guarded by try/catch in existing code. Keep this guard.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Detecting keyboard open/close | Custom heuristics comparing viewport sizes | `visualViewport.resize` event | Standard, fires reliably cross-platform |
| Terminal refit after keyboard | Custom resize calculations | `fitAddon.fit()` | FitAddon handles all edge cases (line height, padding, etc.) |
| Terminal dimension notification | Manual cols/rows calculation | Let `terminal.onResize` do it after `fit()` | Already wired in TerminalView.tsx L272-274 |

**Key insight:** `fitAddon.fit()` → triggers `terminal.onResize` → calls `sendResize(cols, rows)` — this chain is already implemented. The only missing link is triggering `fit()` from the `visualViewport` resize event.

## Common Pitfalls

### Pitfall 1: `window.resize` Misses iOS Keyboard Events
**What goes wrong:** On iOS Safari, opening the keyboard triggers `visualViewport.resize` but NOT `window.resize`. This is the primary bug.
**Why it happens:** iOS Safari distinguishes between layout and visual viewport. `window.resize` only fires when the layout viewport changes.
**How to avoid:** Listen to BOTH `window.resize` (for desktop/Android) AND `visualViewport.resize` (for iOS).
**Warning signs:** Terminal content is invisible or clipped on iPhone/iPad after keyboard opens; Android works fine.

### Pitfall 2: Calling `fit()` Before CSS Transition Completes
**What goes wrong:** `fit()` measures the container at its old size, calculates wrong rows/cols, then the CSS transition completes and the terminal is mismatched.
**Why it happens:** The `visualViewport.resize` event fires once at the START of the keyboard animation on some devices, not at the end.
**How to avoid:** Debounce `fit()` by 100-150ms. iOS keyboard animation is ~250ms; 100ms debounce ensures the container has reached its final size.
**Warning signs:** Terminal briefly shows correct size, then jumps to wrong size.

### Pitfall 3: MobilePromptSheet `fixed bottom-0` Hidden on iOS
**What goes wrong:** The prompt sheet sits below the keyboard on iOS because `fixed` anchors to the layout viewport, not the visual viewport.
**Why it happens:** iOS doesn't shrink the layout viewport when keyboard opens.
**How to avoid:** Use `window.visualViewport` to calculate and set `bottom` offset dynamically.
**Warning signs:** On Android it works; on iOS the prompt sheet is behind the keyboard.

### Pitfall 4: FitAddon Collapse Bug (Known Issue)
**What goes wrong:** Rapid repeated `fit()` calls can cause terminal width to collapse to 1 column (xterm.js issue #5320, March 2025).
**Why it happens:** FitAddon race condition during rapid resize events.
**How to avoid:** Debounce `fit()` calls — the 100ms debounce prevents this scenario.

### Pitfall 5: `pb-12` on Terminal Container
**What goes wrong:** `App.tsx` line 282 has `pb-12 lg:pb-0` on the terminals view container. This 48px bottom padding reserves space for the MobilePromptSheet overlay. If MobilePromptSheet positioning changes, this padding may need adjustment.
**Why it happens:** The terminal content area has padding to avoid being obscured by the fixed prompt sheet.
**How to avoid:** Keep `pb-12` in sync with MobilePromptSheet's collapsed height.

## Code Examples

### Complete fix for TerminalView.tsx resize handling

```typescript
// Source: verified pattern from MDN + xterm.js community
// Replace the existing handleWindowResize block in TerminalView.tsx

// --- NEW: Combined resize handler ---
let resizeDebounceTimer: ReturnType<typeof setTimeout> | null = null;

const refitTerminal = () => {
  if (resizeDebounceTimer) clearTimeout(resizeDebounceTimer);
  resizeDebounceTimer = setTimeout(() => {
    requestAnimationFrame(() => {
      try {
        fitAddon.fit();
        // terminal.onResize fires automatically → sendResize(cols, rows) called
      } catch {
        // Container may have zero dimensions during layout transitions
      }
    });
  }, 100);
};

// window.resize: covers desktop, Android Chrome/Firefox
window.addEventListener('resize', refitTerminal);

// visualViewport.resize: covers iOS Safari keyboard open/close
const visualViewport = window.visualViewport;
if (visualViewport) {
  visualViewport.addEventListener('resize', refitTerminal);
}

// --- Cleanup ---
return () => {
  if (resizeDebounceTimer) clearTimeout(resizeDebounceTimer);
  window.removeEventListener('resize', refitTerminal);
  if (visualViewport) {
    visualViewport.removeEventListener('resize', refitTerminal);
  }
  // ... existing cleanup ...
};
```

### MobilePromptSheet visual viewport positioning

```typescript
// Source: MDN VisualViewport API docs pattern
// Add to MobilePromptSheet component

useEffect(() => {
  const viewport = window.visualViewport;
  const sheet = sheetRef.current;
  if (!viewport || !sheet) return;

  const updateBottomPosition = () => {
    // Calculate how far above the layout viewport bottom we need to go
    // to appear at the visual viewport bottom
    const offsetFromLayoutBottom =
      window.innerHeight - (viewport.offsetTop + viewport.height);
    sheet.style.bottom = `${Math.max(0, offsetFromLayoutBottom)}px`;
  };

  viewport.addEventListener('resize', updateBottomPosition);
  viewport.addEventListener('scroll', updateBottomPosition);
  updateBottomPosition(); // Initial call

  return () => {
    viewport.removeEventListener('resize', updateBottomPosition);
    viewport.removeEventListener('scroll', updateBottomPosition);
    sheet.style.bottom = ''; // Cleanup inline style
  };
}, []); // No deps — viewport reference is stable
```

## Open Questions

1. **Does the `pb-12 lg:pb-0` on the terminal container (App.tsx L282) need adjustment?**
   - What we know: It adds 48px padding to leave room for the collapsed MobilePromptSheet. When the keyboard opens and the app height shrinks, this padding remains — potentially reducing visible terminal area more than necessary.
   - What's unclear: Whether users expect the prompt sheet to be visible while the keyboard is open (it would be covered by keyboard anyway).
   - Recommendation: Consider removing `pb-12` when keyboard is detected as open (i.e., when `visualViewport.height < window.innerHeight * 0.75`). This reclaims the space and allows more terminal content to be visible.

2. **Should MobilePromptSheet collapse when keyboard opens?**
   - What we know: When the keyboard opens (e.g., user taps a terminal textarea), both the terminal and the prompt sheet compete for limited visible space.
   - What's unclear: Whether this is a real user scenario or if the keyboard opens via the key toolbar hidden textarea.
   - Recommendation: The key toolbar's paste input (`<textarea>`) could trigger keyboard — investigate whether `autoCapitalize="none"` etc. help avoid accidental keyboard triggers.

3. **xterm.js issue #5320 (width collapses to 1 on rapid resize) — is debounce sufficient?**
   - What we know: The bug occurs with rapid repeated `fit()` calls. 100ms debounce prevents accumulated rapid calls.
   - What's unclear: Whether this issue affects the specific xterm.js version in use.
   - Recommendation: Check installed version, keep debounce, test manually.
